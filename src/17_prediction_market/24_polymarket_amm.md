# 17.24 Polymarket 的 AMM 结构

## 从 LMSR 到订单簿：Polymarket 的架构演进

```
Polymarket 的历史路径:

  2020 — 早期: 使用 AMM（类 LMSR）做市
    → 问题: 高 Gas（Polygon 缓解了一部分），做市资金效率低
    → 好处: 冷启动简单

  2021-2022 — 转向 CLOB（Central Limit Order Book）:
    → 引入链下匹配引擎（Operator）
    → 链上只做 CTF 结算（split/merge/redeem）
    → 做市商用限价单提供流动性

  2023-2024 — 混合模式:
    → 大市场: 做市商挂单 → 订单簿
    → 小市场: 可能仍用 AMM 做初始定价
    → CTF 始终是结算层
```

## 做市商在 Polymarket 中的角色

```
LMSR 模式:
  做市方 = 协议本身（seed 承担损失）
  → 没有外部做市商
  → 「银行roll」= initial_seed

订单簿模式:
  做市方 = 专业做市商（如 Wintermute、Jump 等）
  → 做市商同时挂买单和卖单
  → 赚取买卖价差（spread）
  → 承担存货风险

  做市商的定价逻辑:
    p_YES_bid = 模型概率 - spread/2
    p_YES_ask = 模型概率 + spread/2

    例: 模型认为 YES 概率 60%，spread 2%
      买 YES @0.59
      卖 YES @0.61
      → 如果成交双向平衡，每对赚 0.02

  与 LMSR 的区别:
    LMSR: 价格由公式计算，滑点由 b 决定
    做市商: 价格由人/算法设定，可以任意调整
    LMSR: 不能「暂停报价」
    做市商: 风险太大时可以撤单
```

## YES/NO 代币的二级市场

```
在 Polymarket 中:
  YES 和 NO 是 ERC-1155 代币
  理论上可以在 Uniswap / Curve 上交易
  但主要流动性在 Polymarket 自己的订单簿上

在教学实现中:
  YES 和 NO 是 Position 中的 u64 余额
  不是独立代币 → 不能在 DEX 交易
  只能通过 split/merge/claim 操作

如何升级到可交易代币:
  方案 1: 为每个市场部署 Coin<YES> / Coin<NO>
    → 可在 Sui DEX 上交易
    → 价格由 DEX AMM 决定（不是 LMSR）
    → 工程复杂但产品可能性大

  方案 2: 用 DeepBook 挂单
    → YES/NO 以限价单形式存在
    → 需要自定义 DeepBook 集成
    → 最接近 Polymarket 的架构
```

## 经济等价性

```
无论用 LMSR 还是订单簿，核心等价关系不变:

  P_YES + P_NO = 1（在均衡时）

  如果 P_YES + P_NO < 1:
    套利: 买 YES + 买 NO → split 的反向 → 无风险利润

  如果 P_YES + P_NO > 1:
    套利: split → 卖 YES + 卖 NO → 无风险利润

  这个等价关系:
    LMSR 中: 由 softmax 数学保证（p_i = exp(q_i/b) / Σ）
    订单簿中: 由套利者执行维护（可能有短暂偏离）

  → 机制不同，但经济结果相同
```

## 本章实现的定位

```
本章教学代码是:
  ✅ 完全链上可验证的 LMSR + CTF 实现
  ✅ 展示了预测市场的核心不变量和状态机
  ✅ 可以作为理解商业产品的数学基础

本章教学代码不是:
  ❌ Polymarket 的替代品
  ❌ 商业级订单簿系统
  ❌ 可以直接上主网的产品

从教学代码到商业产品的距离:
  → 需要链下匹配引擎
  → 需要专业做市商集成
  → 需要 ERC-1155 / Coin 类型的条件代币
  → 需要合规框架
  → 需要外部审计
```
