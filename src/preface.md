# 前言：为什么现在要写 Sui DeFi

DeFi 已经走过了草根实验期。2024 年，链上锁仓价值重新回到千亿美元量级，协议间 composability（可组合性）成为默认假设而非新奇特性。与此同时，新一代公链正在改变 DeFi 的底层执行约束——Sui 是其中技术辨识度最高的一条。

本书不做链的布道，也不做项目的背书。它试图回答一个工程问题：**当底层执行环境从 EVM 的账户模型变为 Sui 的对象模型时，DeFi 协议的设计、实现和安全假设会发生什么变化？**

## 三个写作原则

### 原则一：机制优先

本书的讲解单位不是项目，而是机制。我们不讨论"某某协议好不好"，而是拆解"恒定乘积做市商的价格曲线如何决定 LP 的损益分布"、"CDP 的清算参数如何在极端行情下触发级联效应"。每个机制我们都从数学公式开始，以 Move 代码结束，中间是推演过程。

理解机制意味着你能判断一个协议的行为边界——它在什么条件下正常运作，在什么条件下会失效。这种判断力比知道"哪个 TVL 最高"有用得多。

### 原则二：Sui 视角优先

市面上大多数 DeFi 书籍以 Solidity 和 EVM 为默认环境。本书始终以 Sui 和 Move 为第一视角。每次讲完通用原理，我们立刻追问：**在 Sui 的对象模型下，这个机制的实现有何不同？安全边界发生了什么位移？**

例如：

- Owned Object 的独占访问如何改变 DEX 的撮合路径？
- `drop` ability 的缺失如何保证铸造的 CDP 稳定币不会被意外丢弃？
- PTB（Programmable Transaction Blocks）如何让原子化套利从多笔交易压缩为一笔？

### 原则三：风险优先

本书每讲一个机制，必跟一个风险分析。"警惕"二字不是修辞，而是方法论。我们将风险分解为四个层次：

1. **实现风险**：代码层面的漏洞（溢出、重入、逻辑错误）
2. **机制风险**：设计层面的结构缺陷（预言机延迟、清算参数不当）
3. **系统性风险**：协议间的耦合与传染（一个协议的清算导致另一个协议的坏账）
4. **治理风险**：权限滥用与参数操控

全书最后一篇专门讨论攻击模式与工程防护，但风险意识贯穿每一章。

## Sui 对象模型为 DeFi 带来的独特优势

传统 EVM 架构用账户存储状态，所有状态变更串行执行。A 在 Uniswap 上交易 SUI/USDC，B 在同一个池子的交易必须排队等待。Sui 的对象模型从根本上改变了这个约束：

- **每个资产是独立对象**，拥有全局唯一 ID
- **Owned Object 不需要共识**——如果你只操作自己拥有的对象，验证节点可用快速路径直接确认
- **Shared Object 支持并行**——多个交易可以并行访问不同的共享对象
- **对象可以嵌套**——LP Token 可以作为子对象，保持与父对象的关系

```move
public struct LiquidityPool has key {
    id: UID,
    coin_a: Coin<SUI>,
    coin_b: Coin<USDC>,
    fee_rate: u64,
}

public struct LPToken<phantom A, phantom B> has key, store {
    id: UID,
    amount_a: u64,
    amount_b: u64,
}

public struct AdminCap has key, store {
    id: UID,
    pool_id: ID,
}
```

三种对象的分工：`LiquidityPool` 是共享状态（所有交易者与之交互），`LPToken` 是用户资产（可以自由转移），`AdminCap` 是权限凭证（持有者可以修改费率）。这种分层在 EVM 中需要通过映射表和修饰符实现，在 Sui 中直接由对象模型表达。

> 风险提示：对象模型的灵活性也意味着权限设计需要格外谨慎。一个错误放置的 `store` ability 可能导致资产被转移到非预期的地址。本书将在第2章详细讨论 ability 对安全的影响。

## 本书的局限

本书不讨论投资策略，不提供财务建议。所有收益数字均为示例，不构成任何承诺。代码示例为教学目的简化，未经审计，不可直接用于生产环境。
