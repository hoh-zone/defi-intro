# 4.25 为什么需要订单簿

AMM 是 DeFi 中最流行的交易模型，但它有根本性的局限。订单簿（Orderbook）在许多场景下是 AMM 无法替代的。

## AMM 做不到的事

### 1. 限价单

```
AMM: "我要用 1000 USDC 买 SUI"
  → 立即执行，获得当前池价对应的 SUI
  → 无法指定"我只在 SUI < $1.80 时买入"

Orderbook: "我挂一个限价买单：1000 USDC，限价 $1.80"
  → 如果市场价格达到 $1.80，自动成交
  → 如果没达到，挂单等待
  → 精确控制成交价格
```

### 2. 价格时间优先

```
AMM: 先来后到，所有人面对相同价格
  → 无法奖励提供流动性的 Maker

Orderbook: 价格优先，时间优先
  → 出价最好的订单先成交
  → 同价的先挂单先成交
  → Maker（挂单者）通常享受更低手续费
```

### 3. 无 IL 的做市

```
AMM LP: 必须同时持有两种代币 → 承受无常损失

Orderbook Maker: 挂买卖双边单 →
  买单价: $1.95, 卖单价: $2.05
  每笔成交赚 $0.10 价差
  不持有净头寸 → 无常损失为零

对于专业做市商，Orderbook 比 AMM 更友好
```

## AMM vs Orderbook 全面对比

| 维度 | AMM | Orderbook |
|------|-----|-----------|
| 价格发现 | 算法定价（储备比例） | 市场定价（买卖挂单） |
| 流动性来源 | LP 存入代币 | Maker 挂单 |
| 执行方式 | 即时成交 | 撮合成交 |
| 限价单 | 不支持 | 原生支持 |
| 无常损失 | LP 承受 | Maker 不承受 |
| 做市门槛 | 低（任何人可 LP） | 高（需要策略） |
| 长尾资产 | 友好（只需两种代币） | 困难（需要足够多的挂单） |
| 大额交易 | 滑点大 | 取决于深度 |
| Gas 成本 | 低（简单计算） | 高（撮合逻辑） |

## 什么时候 Orderbook 更好

### 场景一：高流动性主流币对

```
SUI/USDC:
  AMM: 大额交易滑点 0.5-2%
  Orderbook: 如果深度足够，大额交易滑点 0.1-0.5%

  → 交易量大的主流币对，Orderbook 更便宜
```

### 场景二：专业交易者

```
专业交易者需要:
  - 限价单（精确控制成交价格）
  - 止损单（自动平仓）
  - 市价单（即时成交）
  - 批量操作（多个订单管理）

AMM 只能提供市价单，无法满足专业需求
```

### 场景三：价差套利

```
做市商策略:
  在 $1.99 挂买单，在 $2.01 挂卖单
  每笔成交赚 $0.02（0.01% 价差）

AMM 中无法实施这种策略:
  → LP 只能被动接受所有交易
  → 无法选择在什么价格提供流动性

CLMM 部分解决了这个问题（选择价格区间）
但仍然不如 Orderbook 灵活
```

## DeepBook：Sui 的原生 Orderbook

DeepBook 是 Sui 基金会支持开发的链上订单簿 DEX。

### 为什么 Sui 需要 Orderbook

```
Sui 生态发展需要专业级交易基础设施:
  1. 机构交易者需要限价单
  2. 做市商需要 Maker/Taker 模型
  3. 衍生品协议需要精确的价格数据
  4. 聚合器需要 Orderbook 作为流动性来源
```

### DeepBook 的技术优势

```
Sui 对 Orderbook 的天然优势:

1. 低延迟:
   传统链 Orderbook: 交易确认 2-12 秒
   DeepBook: 交易确认 < 1 秒

2. 并行执行:
   不同交易对的撮合可以并行
   不会因为一个热门交易对阻塞其他交易

3. 对象模型:
   每个订单可以是独立对象
   订单管理更灵活

4. PTB 组合:
   限价单 + 闪电贷 + 其他操作
   在一个原子交易中完成
```

## AMM + Orderbook 共存的生态

```
实际的 DEX 生态:

新手用户:
  → 使用 AMM（简单、无需理解挂单）
  → 小额即时交易

专业交易者:
  → 使用 Orderbook（限价单、精确定价）
  → 大额交易、做市

套利者:
  → 同时使用 AMM 和 Orderbook
  → 在两者之间搬砖获利
  → 使两个系统的价格保持一致

聚合器:
  → 同时搜索 AMM 和 Orderbook 的最优价格
  → 为用户提供最佳执行
```

> DeepBook 的具体实现见 4.27 节。Hybrid DEX（AMM + Orderbook 融合）见 4.28 节。
