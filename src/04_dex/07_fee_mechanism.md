# 4.7 Fee 机制设计

手续费是 DEX 经济模型的基石。它决定了 LP 的收入、协议的运营资金、以及交易者的成本。

## 三种手续费角色

```
交易者支付 0.3% 手续费
          │
          ├── LP Fee（如 0.25%）
          │   → 分配给所有 LP
          │   → 存留在池中，增加 LP 份额价值
          │
          └── Protocol Fee（如 0.05%）
              → 分配给协议金库
              → 用于审计、运营、生态建设
```

## LP Fee

LP Fee 是流动性提供者的核心收入来源。

### Fee 如何分配给 LP

LP 不需要"领取"手续费。手续费直接留在池中，增加池的储备量。由于每个 LP 的份额不变但池的总价值增加了，每个份额的价值就自动增加了。

```
交易前:
  reserve_a = 1000, reserve_b = 2000
  total_shares = 1414
  每份价值 = (1000 × $2 + 2000 × $1) / 1414 = $2.83

一笔交易支付 3 单位 A 作为手续费:
  reserve_a = 1003, reserve_b = 2000
  total_shares = 1414（不变）
  每份价值 = (1003 × $2 + 2000 × $1) / 1414 = $2.84

  每份价值增加 = $2.84 - $2.83 = $0.01
  → LP 的手续费收入体现在份额增值中
```

### Sui 中的 LP Fee 实现

```move
// 手续费从输入金额中扣除
let fee_bps = 30; // 0.3%
let amount_in_with_fee = (amount_in as u128) * (10000 - (fee_bps as u128));
// amount_in_with_fee / 10000 就是实际用于交换的金额
// fee = amount_in * fee_bps / 10000 就是手续费
```

LP Fee 不需要单独分配——它自然地留在池中。

## Protocol Fee

协议费是从总手续费中截留的一部分，用于协议运营。

### Protocol Fee 的计算方法

#### 方法一：基于 k 增长（Uniswap V2 风格）

```
每次添加/移除流动性时：
  1. 计算 k_current = reserve_a × reserve_b
  2. 比较 k_current 与 k_last
  3. k 的增长来自交易手续费
  4. 将部分增长铸造为协议份额

fee_shares = total_supply × (sqrt(k_current) - sqrt(k_last))
             × protocol_fee_fraction
             / (sqrt(k_current) × (1 - protocol_fee_fraction))
```

> 完整实现见 `src/04_dex/code/uniswap_v2/sources/pool.move` 中的 `mint_protocol_fee` 函数。

#### 方法二：直接从手续费中扣除

```move
// 更直观的方法
let total_fee = amount_in * fee_bps / 10000; // 0.3%
let protocol_cut = total_fee * protocol_fee_bps / 10000; // 5% 的 0.3%
let lp_cut = total_fee - protocol_cut; // 95% 的 0.3%

// lp_cut 留在池中
// protocol_cut 存入协议金库
```

## Fee 分配

### 分配比例设计

| 阶段     | LP 分配 | 协议分配 | 原因                      |
| -------- | ------- | -------- | ------------------------- |
| 冷启动期 | 90-100% | 0-10%    | 吸引 LP，协议暂不需要收入 |
| 增长期   | 80-85%  | 15-20%   | 协议需要运营资金          |
| 成熟期   | 70-80%  | 20-30%   | 协议需要持续投入安全      |

### 动态费率

高级 DEX（如 Cetus）使用动态费率：

```
基础费率: 0.05%
波动性调整: ±0.01-0.05%

当交易量大、波动性高时 → 提高费率（补偿 LP 风险）
当交易量小、波动性低时 → 降低费率（吸引交易者）
```

## Fee 对各参与方的影响

### 对 LP 的影响

```
LP 年化收益 = (日手续费收入 / 池 TVL) × 365

例：
  池 TVL = $1,000,000
  日交易量 = $500,000
  费率 = 0.3%
  日手续费 = $1,500

  年化 = ($1,500 / $1,000,000) × 365 = 54.75%
```

注意：这是理论收益，实际收益需要扣除无常损失。

### 对交易者的影响

```
实际交易成本 = 名义费率 + 滑点 + Gas

例：
  Swap $10,000 的 SUI → USDC
  名义费率: $30 (0.3%)
  滑点: $50 (0.5%)
  Gas: $0.01

  总成本: $80.01 (0.8%)
```

### 对协议的影响

```
协议年收入 = 总交易量 × 协议费率

例：
  总交易量 = $10亿/年
  协议费率 = 0.05%
  协议年收入 = $500,000

用途：审计费用、Bug Bounty、开发、生态基金
```

## Fee 机制选择指南

| 需求       | 推荐 Fee 设计              |
| ---------- | -------------------------- |
| 简单实现   | 固定费率，LP 费 + 协议费   |
| 精确控制   | 直接从手续费中扣除协议费   |
| 资本效率   | 动态费率（根据波动性调整） |
| LP 友好    | 高 LP 分配比例             |
| 协议可持续 | 合理的协议费比例           |

> 完整的费率实现见 `src/04_dex/code/uniswap_v2/sources/pool.move`。
