## 决策矩阵

### 按交易对类型选择

```
问题: 你的 DEX 要支持什么类型的交易对？

稳定币对（USDC/USDT）:
  → StableSwap
  → 原因: 极低滑点，稳定币价格应该接近 1:1

主流币对（SUI/USDC, ETH/USDC）:
  → CLMM
  → 原因: 资本效率高，流动性集中
  → 参考: Cetus, Turbos

长尾资产（新项目代币）:
  → CPMM 或 CLMM（宽区间）
  → 原因: 简单实现，LP 门槛低
  → 流动性不需要很深

高交易量对（需要专业交易）:
  → Orderbook 或 Hybrid
  → 原因: 限价单、精确定价、做市商友好
  → 参考: DeepBook
```

### 按目标用户选择

```
问题: 你的主要用户是谁？

散户/新手:
  → CPMM 或 CLMM
  → 简单的 Swap 界面
  → 不需要理解限价单

专业交易者:
  → Orderbook 或 Hybrid
  → 限价单、止损单
  → 低手续费

做市商:
  → Orderbook（Maker 返还）或 CLMM（区间做市）
  → 需要精确控制价格和数量

协议开发者:
  → 选择最适合你协议需求的类型
  → 如果需要价格数据 → CPMM 或 CLMM（Oracle 集成好）
```

### 按开发资源选择

```
问题: 你的团队有多少开发资源？

1-2 人团队:
  → CPMM（最简单，代码量最少）
  → 参考 uniswap_v2 实现（本章代码）

3-5 人团队:
  → CLMM 或 StableSwap
  → 中等复杂度，但已经有成熟参考实现

5+ 人团队:
  → DLMM、Orderbook 或 Hybrid
  → 高复杂度，需要大量开发和测试
```

## Sui 特殊考量

### Sui 对象模型的影响

```
选择 CLMM 时:
  → 每个仓位是独立对象 → 天然适合
  → 不需要额外的 NFT 合约

选择 Orderbook 时:
  → 每个订单是独立对象 → 天然适合
  → 动态字段存储价位 → 按需分配

选择 CPMM 时:
  → 对象模型不是关键优势（但也不差）
  → 简单的 shared object 即可
```

### 并行执行的影响

```
所有 DEX 类型都受益于并行执行:
  → 不同交易对的交易可以并行处理
  → 对多池 DEX 吞吐量有巨大提升

对 Orderbook 额外受益:
  → 不同交易对的撮合可以并行
  → Gas 战争只在同一交易对内发生
```

### PTB 的影响

```
Hybrid DEX 在 Sui 上特别有优势:
  → AMM Swap + Orderbook 撮合可以原子化
  → 不需要中间合约
  → Gas 成本更低

多跳路由在 Sui 上更实用:
  → PTB 使得复杂路由的 Gas 成本可接受
  → 原子性保证用户体验
```

## 案例研究

### Cetus：为什么选择 CLMM

```
Sui 生态定位: 主流 DEX，需要服务所有用户

选择 CLMM 的原因:
  1. 资本效率高 → 吸引 LP
  2. 灵活的费率层级 → 服务不同交易对
  3. Oracle 集成 → 为其他协议提供价格
  4. 社区对 Uniswap V3 模型熟悉

不做 Orderbook 的原因:
  - 开发资源集中
  - Sui 已经有 DeepBook 做订单簿
  - 协作 > 竞争
```

### DeepBook：为什么选择 Orderbook

```
Sui 生态定位: 专业交易基础设施

选择 Orderbook 的原因:
  1. Sui 低延迟 → 链上 Orderbook 可行
  2. 机构交易者需要限价单
  3. 与 AMM 互补而非竞争
  4. Sui 基金会支持 → 战略定位

不做 AMM 的原因:
  - 已有 Cetus 等协议覆盖 AMM 需求
  - 专注于 Orderbook 可以做得更好
```

### FlowX：为什么选择 Hybrid

```
Sui 生态定位: 一站式交易平台

选择 Hybrid 的原因:
  1. 同时服务散户和专业交易者
  2. AMM 提供基础流动性
  3. Orderbook 提供精确交易
  4. 差异化竞争

挑战:
  - 开发复杂度最高
  - 需要同时维护两套系统
```

## 总结

```
没有"最好的"DEX，只有"最适合的"DEX

选择流程:
  1. 明确目标用户和交易对
  2. 评估团队开发资源
  3. 考虑 Sui 特有的技术优势
  4. 参考现有协议的选择和经验
  5. 从简单开始，逐步迭代

本章 30 节的核心收获:
  - DEX 的本质是定价机制
  - 不同定价机制适合不同场景
  - Sui 的技术栈为所有类型的 DEX 提供了独特优势
  - 实际实现时，从 CPMM 开始是最佳的学习路径
```
