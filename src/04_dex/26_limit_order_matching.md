# 4.26 限价单撮合引擎

撮合引擎是 Orderbook DEX 的核心。本节详细分析限价单撮合的逻辑。

## 订单类型

### 限价单（Limit Order）

```
买单: "我要在价格 ≤ $1.80 时买入 1000 SUI"
卖单: "我要在价格 ≥ $2.10 时卖出 500 SUI"

特点:
  - 指定了价格和数量
  - 可能不会立即成交（等待市场价格到达）
  - 未成交的部分挂在订单簿上
```

### 市价单（Market Order）

```
"立即买入 1000 SUI，不管价格多少"

特点:
  - 不指定价格，只指定数量
  - 立即与订单簿上的最优价格成交
  - 等价于限价买单，价格为无穷大
```

## 买卖盘与价差

```
订单簿状态:

卖盘（Ask）:
  $2.10 × 500    ← 最低卖价（Best Ask）
  $2.15 × 300
  $2.20 × 200

买盘（Bid）:
  $2.00 × 800    ← 最高买价（Best Bid）
  $1.95 × 600
  $1.90 × 400

价差（Spread）= Best Ask - Best Bid = $2.10 - $2.00 = $0.10 (5%)

中间价（Mid Price）= ($2.10 + $2.00) / 2 = $2.05
```

## Maker vs Taker

| 角色 | 行为 | 手续费 | 贡献 |
|------|------|--------|------|
| Maker | 挂单（提供流动性） | 低（或返还） | 增加订单簿深度 |
| Taker | 吃单（消耗流动性） | 高 | 消耗订单簿深度 |

```
Maker 例子:
  当前 Best Ask = $2.10
  用户挂卖单 $2.15 → 成为 Maker（增加了卖盘深度）

Taker 例子:
  当前 Best Ask = $2.10
  用户市价买入 → 吃掉 $2.10 的卖单 → 成为 Taker
```

Maker 手续费更低的经济学原因：
- Maker 提供流动性 → 对所有交易者有益 → 应该被奖励
- Taker 消耗流动性 → 应该支付更高的成本

## 撮合算法

### 价格-时间优先（Price-Time Priority）

```
规则:
  1. 价格优先: 更优的价格先成交
     买单: 价格高的优先
     卖单: 价格低的优先
  2. 时间优先: 同价格的订单，先到的先成交
```

### 撮合流程

```
新卖单到达: "卖出 700 SUI，限价 $2.00"

当前买盘:
  $2.00 × 500  (Order #1, 最早)
  $2.00 × 300  (Order #2)
  $1.95 × 600  (Order #3)

撮合步骤:
  1. 检查卖价 $2.00 ≤ Best Bid $2.00 → 可以成交
  2. 与 Order #1 成交: 500 SUI × $2.00 = $1,000
     剩余: 700 - 500 = 200 SUI
  3. 与 Order #2 成交: 200 SUI × $2.00 = $400
     剩余: 200 - 200 = 0 SUI
  4. 全部成交，卖单完成

结果:
  Order #1: 完全成交（500 SUI）
  Order #2: 部分成交（200/300 SUI），剩余 100 SUI 继续挂单
  Order #3: 未成交
```

### ASCII 撮合流程图

```
新订单到达
    ↓
检查是否可以与对手盘成交
    ↓
 ┌──Yes──────────────No──┐
 ↓                      ↓
按价格-时间优先撮合    挂入订单簿
 ↓                      ↓
更新剩余量            等待未来成交
 ↓
还有剩余？
 ↓
Yes → 继续匹配下一个对手单
No  → 完成
```

## 部分成交处理

```
场景：卖单 1000 SUI，但买盘只有 700 SUI 的量

方案一：立即成交 700，剩余 300 挂单
  → 大多数 Orderbook 的默认行为

方案二：全部或无（Fill or Kill）
  → 要么全部成交，要么全部取消
  → 适合需要精确数量的交易者

方案三：立即成交或取消（Immediate or Cancel）
  → 成交 700，剩余 300 取消（不挂单）
  → 适合只想立即成交的交易者
```

## Sui 上的订单管理

### 订单作为对象

```move
public struct Order has key {
    id: UID,
    owner: address,
    is_bid: bool,        // true=买单, false=卖单
    price: u64,          // 限价
    quantity: u64,       // 剩余数量
    filled: u64,         // 已成交数量
    created_at: u64,     // 创建时间（用于时间优先）
}
```

Sui 对象模型的优势：
- 每个订单是独立对象 → 可以单独转移、取消
- 订单状态更新不需要全局锁
- 不同交易对的撮合可以并行

### Gas 考虑

```
撮合引擎的 Gas 成本:
  - 单次撮合: O(1) ~ O(n)（取决于需要遍历多少价位）
  - 挂单: O(1)
  - 取消: O(1)

优化策略:
  1. 限制单次撮合的最大价位数
  2. 使用 Merkle 树或跳表存储价位
  3. 批量撮合（多个订单一起处理）
```

## 撮合引擎的公平性

```
问题：如何确保撮合公平？

前端运行风险:
  攻击者看到大额买单 → 提前买入 → 再卖给买单持有者

Sui 的缓解措施:
  1. 低延迟（< 1s 确认）→ 减少前端运行窗口
  2. PTB 原子操作 → 可以组合挂单和取消
  3. 颐和/私有内存池 → 保护大额订单

完全消除前端运行仍然是开放问题
```
