# 22.19 上线前安全流程（Launch Checklist）

这是本章的最后一节，也是一个实用的总结。无论你的协议经过了多少轮审计、多少次测试，上线前的最后一道关卡是这个检查清单。

> **与第 20.4 节的关系**：**合约/经济/运营/文档**四维「上线前最终检查」以 **20.4** 为主表；本节从**风控与分阶段放量**角度补充 Phase 与监控项，避免与 20.4 **逐条重复**——两处对照使用即可。

## Phase 1：测试网验证

### 功能测试

- [ ] 所有公开接口按预期工作
- [ ] 边界条件已测试（零值、最大值、空向量）
- [ ] 所有错误路径已触发并验证
- [ ] 多用户并发操作正常
- [ ] 大额和小额操作都正确

### 压力测试

- [ ] 模拟 TVL 达到目标的 10 倍
- [ ] 模拟价格闪崩 30%/50%/80%
- [ ] 模拟大量用户同时操作
- [ ] 模拟 Gas 价格飙升 50 倍
- [ ] 模拟预言机延迟/失效

### 安全测试

- [ ] 尝试预言机操纵攻击
- [ ] 尝试闪电贷攻击
- [ ] 尝试治理攻击
- [ ] 尝试权限提升
- [ ] 尝试重入攻击（Move 通常免疫，但仍需验证）
- [ ] 尝试整数溢出

## Phase 2：审计（与第 20 章对齐）

**避免与第 20 章重复**：审计节奏、交付物与「上线前最终检查」的**主清单**在 **20.1–20.4**；本节只保留上线视角的**验收勾选项**。

- [ ] 审计轮次与 TVL/风险匹配（策略思路见 **22.5**，勿与 20.4 重复造表）
- [ ] 至少一家熟悉 Sui/Move 的审计方参与；Critical/High 已闭环或已文档化风险接受
- [ ] 修复经审计方确认；未修复项的影响已评估

## Phase 3：Bug Bounty

### 平台选择

- [ ] 选择适合的平台（Immunefi、Code4rena、Sherlock 等）
- [ ] 奖金分级明确
- [ ] 范围定义清晰（哪些合约在范围内）
- [ ] 响应时间承诺合理

### 奖金设置

| 等级 | 奖金范围 | 影响定义 |
|------|---------|---------|
| Critical | $50K-$500K | 直接资金损失 |
| High | $10K-$50K | 参数被非授权修改 |
| Medium | $5K-$10K | 非关键功能异常 |
| Low | $1K-$5K | Gas 优化、信息泄露 |

### 运营准备

- [ ] 有专人负责 Bug Bounty 响应
- [ ] 24 小时内确认报告
- [ ] 7 天内完成 Critical 修复
- [ ] 有紧急联系人

## Phase 4：参数与配置

### 风险参数

- [ ] 最大 LTV 已设置且保守
- [ ] 清算阈值已设置且有足够缓冲
- [ ] Close Factor 已设置（推荐 50%）
- [ ] 清算折扣已设置（推荐 8-10%）
- [ ] 储备因子已设置（推荐 10-15%）
- [ ] 利率拐点已设置（推荐 80%）
- [ ] 最大利率已设置（推荐 100-150%）

### 预言机配置

- [ ] 预言机地址已验证
- [ ] 价格有效期已设置
- [ ] 偏差阈值已设置
- [ ] 备用预言机已配置
- [ ] TWAP 窗口已设置（如适用）

### 权限配置

- [ ] 管理员多签已设置（推荐 3-of-5 或更高）
- [ ] 时间锁已启用（推荐 48 小时）
- [ ] 紧急暂停权限已配置
- [ ] 升级权限已限制
- [ ] 所有权限持有人已确认身份

## Phase 5：监控与响应

### 链上监控

- [ ] TVL 变化监控（> 10% 变化告警）
- [ ] 利用率监控（> 85% 告警）
- [ ] 大额交易监控（> TVL 的 1% 告警）
- [ ] 清算事件监控
- [ ] 价格异常监控

### 告警系统

- [ ] 多渠道告警（Telegram/Discord/邮件/短信）
- [ ] 告警分级（信息/警告/紧急）
- [ ] 值班制度（7×24 小时有人响应）
- [ ] 告警升级流程（从值班到创始人）

### 紧急响应预案

- [ ] 紧急暂停流程已文档化
- [ ] 紧急联系人列表已更新
- [ ] 紧急多签已准备好
- [ ] 社区公告模板已准备
- [ ] 恢复流程已文档化

## Phase 6：灰度发布

### TVL 上限策略

```
推荐的 TVL 上限递进：

第 1 周: $100K 上限
  → 验证基本功能
  → 观察真实用户行为

第 2-4 周: $1M 上限
  → 增加压力
  → 观察清算机制
  → 收集 Bug Bounty 反馈

第 2-3 月: $10M 上限
  → 吸引更大资金
  → 验证清算人在更高 TVL 下的行为

第 3-6 月: $50M-$100M
  → 逐步放开
  → 持续监控

6 个月后: 取消上限
  → 完全开放
  → 但保留紧急暂停能力
```

### 发布前最终检查

- [ ] 测试网所有测试通过
- [ ] 至少一次审计完成且 Critical 问题已修复
- [ ] Bug Bounty 已上线
- [ ] 所有风险参数已设置
- [ ] 所有权限已正确配置
- [ ] 监控系统已上线
- [ ] 告警系统已测试
- [ ] 紧急响应预案已演练
- [ ] TVL 上限已设置
- [ ] 社区公告已发布
- [ ] 文档已更新
- [ ] 团队成员已就位

## 本章回顾

本章建立了 DeFi 风险控制的全景框架：

1. **风险意识**（22.1-22.3）：理解 DeFi 风控的基础和五大风险来源
2. **技术风险**（22.4-22.6）：合约风险模型、审计策略、升级安全
3. **价格风险**（22.7-22.8）：价格框架、操纵模式、多源与 TWAP 架构
4. **清算风险**（22.9-22.11）：抵押率、清算机制、极端情况处理
5. **流动性风险**（22.12-22.14）：利率曲线、银行挤兑、储备金设计
6. **治理风险**（22.15-22.16）：治理攻击防御和权限体系设计
7. **系统性风险**（22.17）：死亡螺旋模型和抗崩溃设计
8. **经济风险**（22.18-22.19）：激励设计和上线前检查

风险控制不是一次性任务，而是持续的工程。每个协议都应该建立持续的风险评估和改进机制。
