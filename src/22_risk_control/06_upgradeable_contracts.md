# 22.6 可升级合约 vs 不可升级合约

合约可升级性是 DeFi 风控中最纠结的设计选择之一。升级能力让协议可以修复漏洞和迭代功能，但也引入了新的攻击面。

## 升级的本质

"升级"在 DeFi 中的含义是：**在不改变用户地址和资金状态的前提下，替换协议的业务逻辑**。

```
不可升级合约：
  部署 → 代码固定 → 如有漏洞，资金永久丢失

可升级合约：
  部署 → 发现问题 → 升级代码 → 修复漏洞
                        ↑
                   这个过程本身是风险
```

## Sui Move 的升级模型

Sui 的包升级机制比 EVM 的 Proxy 模式更简洁：

### 升级流程

1. 发布新版本包到 Sui
2. 通过 `PackageCap` 授权升级
3. 新版本与旧版本共享同一个包 ID

### 升级兼容性规则

| 变更类型 | 兼容性 | 说明 |
|---------|--------|------|
| 添加新函数 | ✅ 兼容 | 不影响现有接口 |
| 修改函数实现 | ✅ 兼容 | 接口不变，逻辑变化 |
| 添加新结构体 | ✅ 兼容 | 新增类型 |
| 修改公开函数签名 | ❌ 不兼容 | 破坏现有调用者 |
| 删除公开函数 | ❌ 不兼容 | 破坏现有调用者 |
| 修改结构体字段 | ❌ 不兼容 | 破坏现有存储 |

### 升级的风险

**1. 存储兼容性风险**

```move
// V1: 原始结构
public struct Position has key {
    id: UID,
    collateral: Balance<SUI>,
    debt: Balance<USD>,
}

// V2: 新增字段
public struct Position has key {
    id: UID,
    collateral: Balance<SUI>,
    debt: Balance<USD>,
    // 新增：健康因子缓存
    health_factor: u64,  // ← 现有对象的此字段为 0，可能引发逻辑错误
}
```

**2. 逻辑变更风险**

升级改变了核心计算逻辑，但链上已有基于旧逻辑创建的状态。新旧逻辑的不一致可能导致：

- 已有仓位的清算阈值突然变化
- 费用计算结果与用户预期不符
- 参数范围检查通过但实际超出安全范围

**3. 权限风险**

谁能触发升级？如果升级权限控制不当：

```
攻击场景：
  1. 攻击者获取升级权限
  2. 部署恶意新版合约
  3. 将所有用户资金转移给攻击者
```

## 升级权限设计方案

### 方案一：完全不可升级

**适用场景**：简单协议、核心基础设施

```
优点：代码即最终版本，用户有最高确定性
缺点：发现漏洞无法修复，只能迁移
```

### 方案二：多签控制升级

**适用场景**：大多数 DeFi 协议

```
升级需要 N-of-M 多签批准
例如：3-of-5 多签，48 小时时间锁

优点：升级门槛高，单人无法作恶
缺点：紧急修复速度慢
```

### 方案三：治理投票升级

**适用场景**：DAO 治理协议

```
升级需要代币持有者投票通过
投票周期：通常 3-7 天

优点：去中心化程度最高
缺点：响应速度最慢，不适合紧急修复
```

### 方案四：分级升级（推荐）

```
┌────────────────────────────────────────────────┐
│              分级升级策略                         │
├────────────────┬───────────────────────────────┤
│ 变更类型       │ 审批流程                       │
├────────────────┼───────────────────────────────┤
│ 紧急安全修复    │ 多签即时批准 + 事后审计         │
│ 非紧急逻辑变更  │ 多签 + 48h 时间锁 + 审计       │
│ 参数调整       │ 治理投票                       │
│ 新功能添加     │ 治理投票 + 审计 + 测试网验证    │
└────────────────┴───────────────────────────────┘
```

## 升级安全检查清单

每次升级前必须完成的检查：

- [ ] 新版本的公开接口是否向后兼容
- [ ] 现有存储对象是否能正确加载
- [ ] 核心计算逻辑是否与旧版本一致（或有意变更且有文档说明）
- [ ] 是否需要数据迁移脚本
- [ ] 升级权限是否经过适当的审批流程
- [ ] 是否在测试网完整验证
- [ ] 是否有回滚方案
- [ ] 是否通知了用户和审计方

## 实践建议

1. **最小升级原则**：每次升级只做最小变更，避免大范围重构
2. **版本标记**：在代码中明确标记版本号，便于追踪
3. **升级公告**：提前通知社区，说明升级内容和预期影响
4. **灰度发布**：先在测试网验证，再在主网上线
5. **紧急暂停准备**：升级后密切监控，准备好紧急暂停方案
