# 22.17 DeFi 死亡螺旋模型

死亡螺旋（Death Spiral）是 DeFi 中最可怕的风险——一个初始冲击通过正反馈循环不断放大，最终导致系统崩溃。理解死亡螺旋的机制，是设计抗崩溃系统的前提。

## 什么是死亡螺旋

死亡螺旋是一个**自我强化的负循环**：

```
初始冲击（价格下跌/流动性撤出/黑客攻击）
  ↓
协议状态恶化（坏账/利用率飙升/清算压力）
  ↓
用户信心下降（恐慌性提款/抛售）
  ↓
协议状态进一步恶化
  ↓
→ 循环加速，直到崩溃
```

与普通风险事件不同，死亡螺旋的特征是**正反馈**——每个环节都加剧下一个环节。

## 三种死亡螺旋模型

### 模型一：清算级联

最常见的死亡螺旋，发生在借贷协议中：

```
步骤 1: 抵押品价格下跌 10%
步骤 2: 部分仓位进入清算
步骤 3: 清算人出售抵押品 → 抛售压力 → 价格继续下跌
步骤 4: 更多仓位进入清算
步骤 5: DEX 流动性被清算消耗殆尽
步骤 6: 清算获得的代币无法出售 → 清算人退出
步骤 7: 剩余仓位无法被清算 → 坏账激增
步骤 8: 协议资不抵债

加速因子:
  - 抵押品集中度（多数用户使用同一种抵押品）
  - 清算效率（出售速度越快，价格冲击越大）
  - DEX 流动性深度（深度越浅，价格冲击越大）
```

### 模型二：稳定币脱锚螺旋

发生在算法稳定币或 CDP 稳定币中：

```
步骤 1: 稳定币价格跌破 $1（如 $0.98）
步骤 2: 用户恐慌，开始大量赎回
步骤 3: 大量赎回 = 大量出售抵押品
步骤 4: 抵押品价格下跌
步骤 5: 更多 CDP 被清算 → 更多抛售
步骤 6: 稳定币价格继续下跌（$0.95 → $0.90 → ...）
步骤 7: 恐慌加剧，所有人同时赎回
步骤 8: 系统崩溃

历史上的案例：
  - UST/Luna (2022): 算法稳定币脱锚 → 死亡螺旋 → $40B 损失
  - Iron Finance (2021): 类似机制
```

### 模型三：关联性螺旋

DeFi 可组合性的阴暗面——风险在协议间传导：

```
步骤 1: 协议 A 出现问题（被攻击或资金损失）
步骤 2: 协议 B 持有协议 A 的资产
  (B 可能是: 使用 A 的 LP 代币作为抵押品的借贷协议)
步骤 3: 协议 B 的资产价值下降 → 清算压力
步骤 4: 协议 C 依赖协议 B 的正常运作
  (C 可能是: 使用 B 的存款凭证作为抵押的杠杆协议)
步骤 5: 协议 C 也开始出问题
步骤 6: 传染扩散到更多协议
步骤 7: 整个 DeFi 生态受影响

关联性矩阵:
  协议 A ←→ 协议 B ←→ 协议 C
     ↕          ↕          ↕
  协议 D ←→ 协议 E ←→ 协议 F
  所有协议互相关联 → 风险传染
```

## 相关性风险

### 为什么分散化在危机中失效

在正常市场中，不同资产的相关性较低，分散化可以降低风险。但在危机中，**所有资产的相关性趋向于 1**：

```
正常市场相关性矩阵:
       SUI    ETH    USDC
SUI    1.0    0.6    0.1
ETH    0.6    1.0    0.1
USDC   0.1    0.1    1.0

危机市场相关性矩阵:
       SUI    ETH    USDC
SUI    1.0    0.9    0.5
ETH    0.9    1.0    0.6
USDC   0.5    0.6    1.0
```

传统金融的经验："在危机中，所有相关性趋近于 1"（所有资产同步下跌）。DeFi 中这一现象更加显著。

### 关联性量化

评估协议的系统性风险暴露：

```
关联性暴露 = Σ (协议持有的其他协议资产价值 × 该协议的风险因子)

风险因子:
  低风险协议（多次审计，TVL > $100M）: 0.1
  中风险协议（1-2 次审计，TVL > $10M）: 0.3
  高风险协议（未审计或新上线）: 0.7

建议: 关联性暴露 < 协议 TVL 的 20%
```

## 抗死亡螺旋设计

### 设计原则一：切断正反馈

```
问题: 清算 → 抛售 → 价格下跌 → 更多清算

解决方案:
  1. 清算抵押品不由自动出售，而是通过荷兰式拍卖
  2. 清算折扣逐渐增加，而非一步到位
  3. 限制单位时间内的清算量
```

### 设计原则二：注入负反馈

```
负反馈机制: 当系统压力增大时，自动引入稳定力

1. 利用率上升 → 利率急剧上升 → 抑制借款、激励存款
2. 价格下跌 → 清算折扣增加 → 吸引更多清算人
3. 流动性下降 → 提款费增加 → 抑制挤兑
4. 系统压力过大 → 熔断器触发 → 暂停操作
```

### 设计原则三：降低关联性

```
1. 限制单一协议资产占总资产的比例
   例: 任何单一协议的资产不超过 TVL 的 10%

2. 使用不同的抵押品类型
   例: 同时支持 SUI、ETH、USDC 作为抵押品

3. 避免使用其他协议的衍生代币作为核心抵押品
   例: 不接受其他借贷协议的存款凭证

4. 分散预言机来源
   例: 不依赖单一预言机
```

### 设计原则四：增加系统缓冲

```
多级缓冲设计:

第 1 级: 利率曲线（缓冲利用率变化）
第 2 级: 清算机制（缓冲价格下跌）
第 3 级: 储备金（缓冲坏账）
第 4 级: 保险基金（缓冲极端损失）
第 5 级: 紧急暂停（缓冲恐慌）
```

## 死亡螺旋压力测试

每个协议都应该模拟以下场景：

| 场景       | 模拟条件             | 通过标准                 |
| ---------- | -------------------- | ------------------------ |
| 主流币闪崩 | ETH/SUI 1 小时跌 30% | 坏账 < 储备金的 50%      |
| 连续清算   | 50% 仓位同时进入清算 | 清算完成率 > 90%         |
| 稳定币脱锚 | USDC 跌至 $0.90      | 协议仍可运作             |
| 跨协议传染 | 关联协议损失 50% TVL | 本协议坏账 < TVL 的 5%   |
| 流动性枯竭 | DEX 深度下降 80%     | 清算仍可完成（可能延迟） |
| 极端组合   | 以上同时发生         | 紧急暂停可阻止崩溃       |

没有协议能完全免疫死亡螺旋，但良好的设计可以将触发阈值提高到极难达到的水平。
