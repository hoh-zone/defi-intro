# 22.8 多源预言机与 TWAP 设计

单一价格来源是 DeFi 的单点故障。本节讨论如何通过多源预言机和 TWAP（时间加权平均价格）来构建更健壮的价格系统。

## 多源预言机设计

### 为什么需要多个数据源

```
单一来源的风险：
  来源 A → 价格
     ↓
  来源 A 出错/被操纵/宕机
     ↓
  整个协议瘫痪

多源设计的韧性：
  来源 A ─┐
  来源 B ─┼→ 聚合器 → 价格
  来源 C ─┘
     ↓
  来源 A 出错 → 被 B、C 覆盖
  来源 A、B 出错 → C 仍可用
```

### 聚合策略

| 策略 | 方法 | 抗操纵性 | 复杂度 |
|------|------|---------|--------|
| 中位数 | 取所有来源的中位数 | 高（忽略极端值） | 低 |
| 截尾均值 | 去掉最高最低后取平均 | 中高 | 中 |
| 加权平均 | 按来源可靠性加权 | 中 | 中 |
| 众数 | 取多数来源一致的值 | 取决于来源数量 | 低 |

中位数是最推荐的策略，因为它天然过滤极端值：

```
来源 A: $1,000
来源 B: $1,005
来源 C: $1,002
来源 D: $50,000  ← 被操纵
来源 E: $998

中位数 = $1,002 ← 极端值被自然忽略
平均值 = $10,601 ← 被极端值严重扭曲
```

### 偏差保护机制

即使使用多源聚合，仍需要偏差保护：

```move
// 偏差保护：当聚合价格与参考价格偏离过大时拒绝使用
fun safe_read_price(
    oracle: &Oracle,
    max_deviation_bps: u64,  // 最大允许偏差（基点）
    reference_price: u64,    // 参考价格（如最近的 TWAP）
): u64 {
    let price = aggregated_price(oracle);
    let deviation = abs_diff(price, reference_price);
    let max_deviation = reference_price * max_deviation_bps / 10000;

    assert!(deviation <= max_deviation, EPriceDeviation);
    price
}
```

偏差保护参数建议：

| 资产类型 | 建议偏差阈值 | 说明 |
|---------|-------------|------|
| 稳定币 | 1-2% (100-200 bps) | 稳定币价格应非常接近 $1 |
| 主流币 (BTC/ETH/SUI) | 5-10% (500-1000 bps) | 正常波动范围 |
| 山寨币 | 15-25% (1500-2500 bps) | 波动性更大 |
| 长尾资产 | 不建议使用预言机 | 直接使用不可靠 |

## TWAP 设计

### 原理

TWAP（Time-Weighted Average Price）不是使用某个时间点的即时价格，而是使用一段时间内的平均价格。这天然过滤了瞬时价格操纵。

```
TWAP 计算原理：

时间段 [t1, t2] 内的 TWAP：
  TWAP = (cumulative_price(t2) - cumulative_price(t1)) / (t2 - t1)

其中 cumulative_price 是价格的累积值：
  cumulative_price(t) = Σ price(t_i) * (t_{i+1} - t_i)
```

### TWAP 窗口选择

| 窗口时长 | 抗操纵性 | 时效性 | 适用场景 |
|---------|---------|--------|---------|
| 1 分钟 | 低 | 极高 | 高频交易 |
| 10 分钟 | 中 | 高 | 清算定价 |
| 30 分钟 | 高 | 中 | 借贷抵押品定价 |
| 1 小时 | 很高 | 低 | CDP 铸造稳定币 |
| 24 小时 | 极高 | 极低 | 长期参考价格 |

窗口选择的核心权衡：**更长的窗口更安全，但可能错过真实的价格变化**。

### TWAP 的攻击成本

使用 TWAP 后，攻击者必须在**整个时间窗口内维持被操纵的价格**，这极大提高了攻击成本：

```
即时价格操纵（无 TWAP）：
  成本 = 一次交易的成本 ≈ 几十美元

10 分钟 TWAP：
  成本 = 在 10 分钟内维持价格的成本
       = 需要持续交易对抗市场力量
       ≈ 数万到数十万美元

30 分钟 TWAP：
  成本 ≈ 数十万到数百万美元
  且面临套利者逆向交易的风险
```

## 推荐的价格架构

```
┌─────────────────────────────────────────────────┐
│              推荐价格架构                         │
│                                                 │
│  Pyth/Supra ──┐                                 │
│  Switchboard ─┤→ 多源聚合器 → 中位数价格         │
│  DEX TWAP ────┘         │                       │
│                         ↓                       │
│                  偏差检测 + 时效检查              │
│                         │                       │
│                         ↓                       │
│                  业务逻辑使用                    │
│                         │                       │
│                         ↓                       │
│                  单笔操作限额                     │
│                         │                       │
│                         ↓                       │
│                  紧急熔断（价格偏离 > 30%）       │
└─────────────────────────────────────────────────┘
```

这个架构在 Sui 上的实现可以参考第 5 章的预言机插槽设计和多预言机聚合内容。

关键要点：
1. **永远不要只使用一个价格来源**——哪怕是最可靠的来源也可能宕机
2. **TWAP 是防御价格操纵的基础设施**——但不是万能的
3. **偏差保护是最后一道防线**——当价格与预期差距过大时，宁可暂停也不要使用错误价格
4. **不同业务场景使用不同的窗口长度**——清算用短窗口，CDP 用长窗口
