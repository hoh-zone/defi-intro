# 22.7 为什么价格是 DeFi 的生命线

价格（Price）是 DeFi 协议运转的核心输入。几乎所有关键决策都依赖价格：是否触发清算、能借多少钱、交易得到多少代币。价格出错，整个系统就出错。

## 价格在 DeFi 中的角色

```
价格 → 借贷：决定抵押率和清算触发点
价格 → DEX：决定交易兑换比率
价格 → 清算：决定何时清算、清算多少
价格 → 衍生品：决定持仓价值和保证金要求
价格 → CDP：决定能铸造多少稳定币
```

一个数字出错，五类协议全部受影响。

## 价格错误的后果分级

| 偏离程度 | 后果 | 影响 |
|---------|------|------|
| < 1% | 轻微套利机会 | 影响小 |
| 1-5% | 清算误触发或漏触发 | 部分用户受损 |
| 5-20% | 大规模错误清算 | 系统性损失 |
| > 20% | 协议经济模型崩溃 | 可能导致协议破产 |

## 价格使用的三个维度

评估协议的价格风险，需要检查三个维度：

### 1. 价格来源：数据从哪里来？

| 来源 | 可靠性 | 延迟 | 操纵成本 |
|------|--------|------|---------|
| DEX TWAP | 中 | 高（需时间窗口） | 中 |
| Chainlink/Pyth | 高 | 低 | 极高 |
| 单一 DEX 现价 | 低 | 极低 | 低（闪电贷即可） |
| 协议自报价格 | 极低 | 无 | 几乎为零 |

### 2. 价格时效：数据有多新鲜？

```
                    安全性
                      ↑
   TWAP(30min) ●      │
                      │
   TWAP(10min)  ●     │
                      │
   即时价格      ●     │
                      │
   过期价格       ●    │
                      └──────────→ 时效性
```

安全性和时效性之间存在权衡。TWAP 更安全但更滞后，即时价格更及时但更容易被操纵。

### 3. 价格验证：数据是否合理？

即使数据来源可靠、时效足够，仍然需要验证价格是否在合理范围内：

```move
// 价格合理性检查
fun validate_price(price: u64, reference: u64, max_deviation_bps: u64): bool {
    if (price == 0) return false;

    let lower = reference * (10000 - max_deviation_bps) / 10000;
    let upper = reference * (10000 + max_deviation_bps) / 10000;

    price >= lower && price <= upper
}
```

## 错价 = 系统性风险

当价格错误时，不同协议的灾难模式：

### 借贷协议（价格被高估）

```
1. 攻击者存入被高估的资产作为抵押
2. 按照虚高的价格借出大量其他资产
3. 价格恢复 → 抵押品价值骤降 → 仓位不足额
4. 如果清算不及时 → 产生坏账
5. 存款人无法全额提款
```

### DEX（价格被操纵）

```
1. 攻击者通过闪电贷大量买入某代币
2. DEX 价格上涨
3. 其他协议使用这个价格做决策 → 错误决策
4. 攻击者获利，协议受损
```

### CDP/稳定币（价格被低估）

```
1. 抵押品价格被人为压低
2. 大量 CDP 仓位被错误清算
3. 清算拍卖以低价出售抵押品
4. 用户遭受不必要的损失
5. 稳定币可能脱锚（恐慌性抛售）
```

## 价格风险评估框架

对任何使用价格的 DeFi 协议，回答以下问题：

1. **单一来源依赖**：协议是否只依赖一个价格来源？如果是，这个来源失效的后果是什么？
2. **操纵成本**：攻击者操纵价格需要多少成本？是否大于潜在收益？
3. **时效要求**：协议对价格更新的频率要求是什么？如果更新延迟 1 分钟、10 分钟、1 小时，分别会发生什么？
4. **极端值处理**：如果价格突然变为 0 或者变为正常的 10 倍，协议会怎么处理？
5. **备用机制**：主价格源失效时，是否有备用方案？
