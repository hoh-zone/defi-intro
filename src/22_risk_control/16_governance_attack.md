# 22.16 治理攻击与闪电贷投票

治理是 DeFi 协议的"操作系统"——控制参数、管理资金、决定升级方向。治理被攻破，整个协议都处于危险之中。

## 治理攻击的类型

### 类型一：闪电贷投票攻击

```
攻击原理：
  治理代币持有者可以投票
  → 闪电贷借入大量治理代币
  → 用借来的代币投票通过恶意提案
  → 投票结束后归还闪电贷

成本：闪电贷手续费（约 0.1%）
收益：提取协议金库、修改参数获利
```

闪电贷投票的攻击路径：

```
1. 创建恶意治理提案
   "将协议金库的 50% 转移到地址 X"

2. 闪电贷借入 1000 万治理代币

3. 用借来的代币投票支持提案
   投票权重 = 1000 万 >> 真实持有者的总票数

4. 提案通过

5. 归还闪电贷

6. 恶意提案执行，资金被转移
```

### 类型二：治理慢攻击

```
攻击原理：
  攻击者逐步积累治理代币
  不触发任何告警阈值
  达到足够权重后发起恶意提案

更隐蔽但更危险：
  攻击者可以通过多个地址分散购买
  避免被识别为单一实体的集中持仓
```

### 类型三：提案钓鱼

```
攻击原理：
  创建看似无害的提案
  实际包含隐蔽的恶意代码

例：
  "优化 Gas 效率" 的提案
  实际修改了提款权限

防范：
  任何提案都需要完整的代码审计
  社区专家审核 + 技术分析报告
```

## 治理攻击的防御机制

### 防御一：投票快照

```
快照机制：
  在提案创建时拍摄代币持有快照
  投票权重基于快照，而非实时余额

效果：
  攻击者在提案创建后购买的代币没有投票权
  闪电贷投票无效（提案创建前借入才有用，
  但提案创建时无法预知会有什么提案）
```

### 防御二：时间锁（Timelock）

```
时间锁机制：
  提案通过后，不立即执行
  需要等待 T 时间（通常 24-72 小时）
  在等待期间，任何人可以取消恶意提案

效果：
  即使攻击者通过了恶意提案
  社区有时间发现并取消
```

### 防御三：法定人数（Quorum）

```
法定人数要求：
  提案通过需要至少 Q% 的代币参与投票

例：
  法定人数 = 20% 的总供应量
  即使攻击者借入 10% 的代币
  还需要 10% 的真实持有者参与投票

效果：
  防止低参与率时少数代币控制治理
```

### 防御四：委托限制

```
委托限制：
  单个地址的最大投票权重有上限
  或投票权重按平方根衰减

效果：
  防止单一实体（或闪电贷借入者）集中过多投票权
```

## 治理参数设计

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| 提案门槛 | 0.1-1% 代币供应量 | 创建提案需要的最低代币数量 |
| 投票期 | 3-7 天 | 投票持续时间 |
| 执行延迟 | 1-3 天 | 通过后到执行的时间锁 |
| 法定人数 | 10-20% | 提案通过的最低参与率 |
| 赞成门槛 | 50-67% | 赞成票需要达到的比例 |
| 取消条件 | 任何时间锁期间可取消 | 紧急取消机制 |

```move
public struct GovernanceProposal has key {
    id: UID,

    end_epoch: u64,
    for_votes: u64,
    against_votes: u64,
    quorum: u64,
    executed: bool,
    // 提案内容...
}
```

### Capability 模式在治理中的应用

```move
// 紧急治理：多签控制
public struct EmergencyCap has key, store {
    id: UID,
    // 限制：只能执行特定类型的操作
}

// 常规治理：代币投票
public struct VoteReceipt has key {
    id: UID,
    proposal_id: ID,
    votes: u64,
    direction: bool,
}
```

## 治理安全检查清单

- [ ] 是否使用投票快照防止闪电贷投票？
- [ ] 是否有时间锁延迟执行？
- [ ] 是否有法定人数要求？
- [ ] 提案门槛是否合理（不会太低也不会太高）？
- [ ] 是否有紧急取消机制？
- [ ] 关键参数修改是否需要更高的投票门槛？
- [ ] 治理代币的分配是否合理（避免过度集中）？
- [ ] 是否有监控异常投票行为的机制？
