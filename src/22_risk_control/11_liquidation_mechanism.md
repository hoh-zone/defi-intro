# 22.11 清算机制与清算人网络

清算机制是 DeFi 借贷协议的"免疫系统"。当用户的抵押品不足以支撑其借款时，清算机制确保系统的偿付能力。清算设计的质量直接决定协议在极端市场条件下的生死。

## 清算的经济模型

### 清算触发条件

```
清算触发条件：

健康因子 = (抵押品价值 × 清算阈值) / 借款价值

当 健康因子 < 1.0 时，仓位可被清算

等价地：
  当 LTV > 清算阈值时，仓位可被清算
```

### 清算奖励

清算人执行清算的动力来自奖励。奖励设计决定了清算人的积极性：

| 奖励类型 | 描述 | 典型值 |
|---------|------|--------|
| 固定折扣 | 清算人以折扣价购买抵押品 | 5-10% |
| 动态折扣 | 折扣随健康因子恶化增加 | 5-15% |
| 代币奖励 | 额外发放治理代币 | 变化大 |
| 利息分成 | 获得被清算仓位的部分利息 | 变化大 |

```
清算奖励设计的目标：
  1. 正常情况下，清算奖励足够覆盖清算成本 + 提供利润
  2. 极端情况下，奖励足够激励清算人竞争（Gas 战争中仍有动力）
  3. 奖励不能过高，否则激励"恶意清算"（故意压低价格触发清算）
```

### Close Factor

Close Factor 限制单次清算可以偿还的债务比例：

```
Close Factor = 单次可清算的最大债务比例

例：
  Close Factor = 50%
  用户借款 1000 USDC
  单次清算最多偿还 500 USDC
  剩余 500 USDC 仍然是债务
```

Close Factor 的作用：
- **过高**（如 100%）：一次清算就关闭整个仓位，对借款人惩罚过重
- **过低**（如 10%）：清算速度太慢，可能来不及阻止坏账产生
- **推荐值**：50%（平衡清算效率和借款人保护）

## 清算人网络

### 清算人的类型

| 类型 | 响应速度 | 成本 | 可靠性 |
|------|---------|------|--------|
| 专用清算 Bot | 毫秒级 | 需要基础设施投入 | 高 |
| MEV 搜索者 | 毫秒级 | 共享收益给矿工/验证者 | 中 |
| 普通用户 | 秒级 | 低 | 低 |
| 协议内清算模块 | 即时 | 无额外成本 | 最高（但受限于 Gas） |

### 清算 Bot 的运行逻辑

```
清算 Bot 工作流：

1. 监控链上所有仓位的健康因子
2. 识别健康因子 < 1.0 的仓位
3. 计算清算收益：
   收益 = 清算奖励 - Gas 费用 - 滑点成本
4. 如果收益 > 0，提交清算交易
5. 将获得的抵押品在市场上出售

关键优化：
  - 预计算即将进入清算状态的仓位
  - 预先准备清算所需的资金
  - 使用最优的 Gas 价格确保交易被打包
```

### Sui 上的清算优势

Sui 的并行执行和 PTB（可编程交易块）为清算提供了独特优势：

```
传统链上清算（串行）：
  交易1: 获取闪电贷
  交易2: 执行清算
  交易3: 出售抵押品
  交易4: 归还闪电贷
  → 4 笔交易，可能跨越多个区块

Sui PTB 清算（原子化）：
  PTB:
    1. 获取闪电贷（同一 PTB）
    2. 执行清算（同一 PTB）
    3. 出售抵押品（同一 PTB）
    4. 归还闪电贷（同一 PTB）
  → 1 笔交易，原子执行，无 MEV 风险
```

## 清算机制设计检查清单

| 检查项 | 要求 |
|-------|------|
| 清算奖励 | 是否足够激励清算人在 Gas 战争中竞争？ |
| Close Factor | 是否在保护借款人和确保清算效率之间取得平衡？ |
| 清算路径 | 是否支持通过 PTB 实现原子化清算？ |
| 部分清算 | 是否允许部分清算（而非强制全额清算）？ |
| 清算人准入 | 是否任何人都可以成为清算人？ |
| 清算失败处理 | 如果无人清算会发生什么？ |
| 紧急清算 | 协议是否有自清算机制作为最后手段？ |

## 清算人激励可持续性

清算人生态系统的健康度直接影响协议安全：

- **清算人数量**：越多越好，避免垄断
- **清算速度**：从仓位进入可清算状态到实际清算的时间
- **清算覆盖率**：被清算的仓位占所有可清算仓位的比例

目标指标：
- 清算延迟 < 30 秒（正常市场）
- 清算覆盖率 > 95%（正常市场）
- 至少 5 个独立清算人活跃运营
