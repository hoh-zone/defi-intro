# 22.14 流动性枯竭与银行挤兑

银行挤兑不是传统金融的专利。DeFi 协议同样面临"挤兑"——当大量用户同时提款时，协议可能无法满足所有提款请求。

## DeFi 银行挤兑的机制

### 传统银行挤兑 vs DeFi 挤兑

| 维度 | 传统银行 | DeFi 协议 |
|------|---------|----------|
| 触发原因 | 传闻、负面新闻 | 链上数据可见，触发更快 |
| 速度 | 几小时到几天 | 几分钟到几小时 |
| 信息透明度 | 低（存款人不知道银行的真实状况） | 高（所有数据链上可见） |
| 央行干预 | 有（最后贷款人） | 无 |
| 结果 | 政府存款保险（FDIC）覆盖部分损失 | 损失通常由存款人承担 |

### DeFi 挤兑的典型流程

```
触发事件（如：协议被黑客攻击的传闻）
  ↓
1. 少数大额存款人开始提款（鲸鱼先跑）
  ↓
2. 利用率快速上升（可借资金减少）
  ↓
3. 利率飙升（存款人担心收益下降或本金损失）
  ↓
4. 更多存款人加入提款
  ↓
5. 利用率接近 100%
  ↓
6. 部分存款人无法提款（资金已被借出）
  ↓
7. 恐慌蔓延，所有人都想提款
  ↓
8. 协议流动性完全枯竭
  ↓
9. 无法提款的存款人承受损失
```

关键差异：DeFi 的链上透明性使得挤兑的**触发速度更快**——所有人都能实时看到协议的资金状况。

## 流动性枯竭的量化分析

### 流动性缺口

```
流动性缺口 = 待提款总额 - 可用资金

待提款总额 = 存款总额
可用资金 = 存款总额 - 借款总额 + 可即时回收的借款

即时流动性 = 存款总额 × (1 - 利用率)

例：
  存款总额: $100M
  利用率: 85%
  即时流动性: $100M × 15% = $15M

  如果有 $20M 要提款 → 缺口 $5M
```

### 流动性风险指标

| 指标 | 计算方法 | 安全阈值 |
|------|---------|---------|
| 即时流动性比率 | (1 - 利用率) | > 20% |
| 流动性覆盖率 | 即时流动性 / 日均提款量 | > 3x |
| 最大可提款金额 | 即时流动性 | > 协议 TVL 的 5% |
| 提款队列深度 | 待处理提款请求数量 | 0（正常情况） |

## DeFi 挤兑的历史案例模式

### 模式一：恐慌性提款

```
触发：协议出现安全传闻
过程：
  1. 链上监控 bot 检测到大额提款
  2. 社交媒体传播
  3. 更多用户跟进提款
  4. 利用率飙升，利率飙升
  5. 未提款的用户面临更高风险
```

### 模式二：杠杆螺旋清算

```
触发：抵押品价格大幅下跌
过程：
  1. 清算发生，借款被强制偿还
  2. 偿还的借款返回存款池 → 短暂增加流动性
  3. 但同时，存款人因恐慌提款
  4. 利用率上升，借款成本飙升
  5. 更多借款人被迫平仓
  6. 连锁反应 → 系统性流动性危机
```

### 模式三：跨协议传染

```
触发：协议 A 出现问题
过程：
  1. 协议 A 被攻击或出现流动性危机
  2. 协议 B 持有协议 A 的资产（LP 代币、存款凭证等）
  3. 协议 B 的资产价值下降
  4. 协议 B 的用户开始提款
  5. 恐慌蔓延到协议 C、D...
```

## 防御挤兑的设计策略

### 策略一：提款排队

```
当利用率 > 阈值时，提款进入排队：
  1. 提款请求被记录
  2. 当有借款人还款时，按排队顺序释放
  3. 排队等待期间仍然获得利息

优点: 防止先到先得的挤兑
缺点: 用户资金被锁定，可能加剧恐慌
```

### 策略二：提款费

```
提款费 = 基础费率 + (利用率 / 利用率上限) × 附加费率

例：
  利用率 < 80%: 提款费 = 0
  利用率 80-90%: 提款费 = 0.5%
  利用率 90-95%: 提款费 = 2%
  利用率 > 95%: 提款费 = 5%

优点: 经济激励抑制挤兑
缺点: 正常提款也可能受影响
```

### 策略三：提款限额

```
单笔提款上限 = 协议即时流动性 × 单笔比例

例：
  即时流动性: $10M
  单笔比例: 5%
  单笔提款上限: $500K

优点: 简单直接
缺点: 大额用户受限
```

### 策略四：时间锁提款

```
提款流程：
  1. 用户发起提款请求
  2. 等待 T 时间（如 24 小时）
  3. 提款可用时自动执行

优点: 给协议缓冲时间
缺点: 用户体验差
```
