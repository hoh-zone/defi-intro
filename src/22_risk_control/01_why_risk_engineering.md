# 22.1 为什么 DeFi 本质是"风险工程"

## 三个不可回避的事实

DeFi 协议的开发者不是在写普通软件，而是在构建**金融基础设施**。这意味着三个不可回避的事实：

### 1. 资金不可逆

传统软件有"回滚"按钮。DeFi 没有。

```move
// 一旦交易上链，资产转移不可撤销
transfer::public_transfer(coin, recipient);
// 没有 undo，没有客服电话，没有法院传票
```

一笔错误的转账、一个被利用的漏洞、一次清算失败——损失是终局的。这要求开发者在部署前就对每一个资金路径做**穷举式的风险分析**。

### 2. 无人工兜底

传统金融有"熔断机制"——股价暴跌时交易所暂停交易，银行挤兑时央行提供流动性，支付出错时客服可以手动调账。

DeFi 的链上协议没有这些：

- 没有客服可以追回资金
- 没有监管可以暂停交易（除非协议自己设计了暂停功能）
- 没有"最后贷款人"在流动性枯竭时兜底

协议必须在代码层面**预先设计所有边界情况的处理方式**。

### 3. 代码即金融基础设施

在传统金融中，金融逻辑由多个系统共同实现：交易所有撮合引擎、银行有核心系统、清算所有结算系统。单个系统的 bug 影响有限，因为有多层冗余。

DeFi 中，一个智能合约就是**完整的金融系统**：

```
传统金融：撮合引擎 + 风控系统 + 清算系统 + 结算系统
    ↕ 多层冗余
DeFi 协议：一个 Move 包（sources/*.move）
    ↕ 单点故障
```

一个函数的错误，可以同时影响定价、清算、提款——所有环节。

## 风险工程的思维转变

从"写代码"到"做风险工程"，需要三个思维转变：

| 传统开发思维            | 风险工程思维                 |
| ----------------------- | ---------------------------- |
| "功能能不能跑通？"      | "功能在极端情况下怎么表现？" |
| "测试覆盖率高不高？"    | "攻击者会怎么利用这个功能？" |
| "出了 bug 打个补丁就好" | "出了 bug 资金就没了"        |
| "先上线再迭代"          | "上线前做穷举式风险分析"     |

## 风险工程的核心问题

每个 DeFi 协议都应该回答四个核心问题：

1. **资金安全边界**：协议中最大可能的资金损失是多少？在什么条件下会发生？
2. **系统韧性**：当外部条件（价格、流动性、网络）急剧恶化时，系统是否能维持运作？
3. **攻击成本**：攻击者需要多少成本才能让系统失效？这个成本是否足够高？
4. **恢复路径**：当风险事件发生后，系统是否有恢复机制？

这四个问题贯穿本章所有内容。每一节的风险分析，本质上都是在回答这四个问题中的某一个或某几个。
