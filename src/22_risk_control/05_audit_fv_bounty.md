# 22.5 审计、形式化验证与 Bug Bounty

代码风险控制的三道防线：审计、形式化验证、Bug Bounty。三者各有边界，不能互相替代。

> **与第 20 章的分工**：**审计准备、权限矩阵、治理与上线前检查清单**以 **20.1–20.4** 为准；本节只回答风控视角的「三道防线各自能证明什么、不能证明什么」以及**投入组合策略**，避免与 20 章、**22.19** 的清单段落重复。

## 审计的边界

### 审计能做什么

- **发现已知模式的问题**：重入、权限错误、整数溢出等
- **验证业务逻辑的正确性**：对照白皮书检查代码实现
- **评估代码质量**：模块化、可读性、可维护性
- **检查最佳实践的遵循度**：事件、错误码、文档

### 审计不能做什么

- **不能保证零漏洞**：审计是抽样检查，不是穷举证明
- **不能覆盖部署后变化**：参数调整、新市场上线可能引入新风险
- **不能替代运行时监控**：很多问题只在特定链上状态才会触发

### 多审计策略

对于 TVL 超过 $1M 的协议，推荐至少两次独立审计：

| 审计轮次 | 时机 | 重点 | 深度 |
|---------|------|------|------|
| 第一轮 | 功能冻结后 | 整体架构和核心逻辑 | 全面扫描 |
| 第二轮 | 第一轮修复完成后 | 修复引入的新问题 + 深入边界情况 | 聚焦难点 |
| 可选第三轮 | 主网上线前 | 最终状态确认 | 全量回归 |

选择审计公司的考量因素：

- **技术专长**：是否熟悉 Move/Sui 生态（而非仅 EVM 经验）
- **行业口碑**：过往审计的协议是否出过问题
- **方法论**：是否使用形式化验证、模糊测试等高级技术
- **沟通质量**：报告是否清晰，是否提供修复建议

## 形式化验证

### 什么是形式化验证

形式化验证（Formal Verification, FV）用数学方法**证明**代码满足特定属性——不是"测试了没发现问题"，而是"数学证明了不存在某类问题"。

### 何时值得做形式化验证

| 场景 | 是否推荐 FV | 原因 |
|------|-----------|------|
| 核心 AMM 曲线 | ✅ 推荐 | 数学公式需要精确验证 |
| 清算引擎 | ✅ 推荐 | 极端情况多，测试难以覆盖 |
| 简单的代币转账 | ❌ 不必要 | 标准模式，常规审计足够 |
| 治理模块 | ⚠️ 视情况 | 逻辑简单但影响大 |

### Move Prover

Sui Move 自带的形式化验证工具 Move Prover 可以验证：

```
// 示例：验证清算函数永远不会产生坏账
// spec 规约：
//   清算后，协议的总资产 >= 总负债
//   清算者获得的抵押品价值 <= 债务价值 + 清算奖励
```

Move Prover 的优势：
- 与 Move 语言原生集成
- 可以证明内存安全和数学属性
- 适合验证不变量（invariants）

Move Prover 的局限：
- 无法验证外部依赖（预言机、其他协议）
- 规约本身的正确性需要人工审查
- 学习曲线陡峭

## Bug Bounty

### Bug Bounty 分级

参考 Immunefi 的标准分级：

| 等级 | 影响 | 典型奖励 |
|------|------|---------|
| Critical | 资金直接被盗取或永久锁定 | $50K - $500K+ |
| High | 协议参数被非授权修改、严重逻辑错误 | $10K - $50K |
| Medium | 非关键资金损失、功能异常 | $5K - $10K |
| Low | 信息泄露、Gas 优化 | $1K - $5K |

### Bug Bounty 的最佳实践

1. **分级明确**：每个等级对应的奖金范围要清晰
2. **响应时间承诺**：Critical 问题 24 小时内确认，7 天内修复
3. **范围清晰**：哪些合约在范围内，哪些不在
4. **持续运营**：不是审计前的短期活动，而是长期的安全投资

## 三道防线的组合策略

```
┌─────────────────────────────────────────┐
│           安全投入的 ROI 曲线             │
│                                         │
│ 投入 ─→  审计  →  形式化验证  →  Bounty  │
│            │         │            │     │
│ 成本       中        高           持续   │
│ 覆盖度     高        最高         广泛   │
│ 信心提升   +40%      +20%         +15%  │
└─────────────────────────────────────────┘
```

推荐的组合策略：

- **TVL < $100K**：1 次审计 + Bug Bounty
- **TVL $100K - $1M**：2 次审计 + Bug Bounty
- **TVL $1M - $10M**：2 次审计 + 核心模块 FV + Bug Bounty
- **TVL > $10M**：2+ 次审计 + 全面 FV + 持续 Bug Bounty + 实时监控

无论投入多少，三道防线都不能提供 100% 的保证。风险管理的目标是**将风险降低到可接受的范围内**，而不是完全消除风险。
