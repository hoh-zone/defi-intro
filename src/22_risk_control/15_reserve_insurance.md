# 22.15 储备金与保险基金机制

当清算失败产生坏账、流动性枯竭导致损失时，谁来承担？储备金和保险基金是协议级的"减震器"。

## 储备因子（Reserve Factor）

### 定义

储备因子是协议从利息收入中截留的比例，用于构建储备金：

```
协议利息收入 = 借款利率 × 借款总额
储备金提取 = 协议利息收入 × 储备因子
存款人收益 = 协议利息收入 × (1 - 储备因子)

例：
  借款总额: $100M
  借款利率: 5%
  储备因子: 10%

  协议利息收入 = $5M/年
  储备金提取 = $500K/年
  存款人收益 = $4.5M/年
```

### 储备因子参数选择

| 资产风险 | 建议储备因子 | 原因 |
|---------|-------------|------|
| 低风险（USDC/USDT） | 5-10% | 价格稳定，坏账概率低 |
| 中风险（ETH/SUI） | 10-15% | 有波动性，需要更多缓冲 |
| 高风险（山寨币） | 15-30% | 波动性大，坏账概率高 |

### 储备金的用途

```
储备金的优先级：

1. 覆盖坏账（最高优先级）
   当清算产生亏损时，用储备金填补

2. 紧急流动性提供
   当流动性枯竭时，释放储备金满足提款

3. 协议运营
   审计费用、Bug Bounty 奖金、开发成本

4. 生态激励
   流动性挖矿奖励、生态基金
```

## 协议金库（Protocol Treasury）

### 与储备金的区别

| 维度 | 储备金 | 协议金库 |
|------|--------|---------|
| 来源 | 利息收入的固定比例 | 治理代币、手续费、投资收益 |
| 用途 | 特定用途（覆盖损失） | 广泛用途（运营、发展） |
| 管理 | 通常是自动的 | 通过治理投票 |
| 透明度 | 链上可查 | 链上可查 |

### 金库的资金来源

```
协议金库的资金来源：

1. 治理代币分配（上线时预留给协议）
   → 占总量 10-30%

2. 手续费收入
   → DEX 的交易手续费、借贷的协议费

3. 清算收入
   → 清算折扣的一部分归协议

4. 战略投资
   → 金库资产的投资收益
```

## 坏账处理机制

### 坏账产生的原因

```
坏账 = 借款人债务 > 抵押品价值

原因：
  1. 清算延迟（价格下跌过快）
  2. 清算奖励不足（无人愿意清算）
  3. Gas 战争（清算成本过高）
  4. 智能合约漏洞（逻辑错误导致错误清算）
  5. 预言机故障（使用错误价格）
```

### 坏账处理的优先级

```
坏账处理优先级（谁先承担损失）：

1. 清算折扣（清算人承受部分损失）
   → 通过提高折扣比例吸收

2. 储备金（协议承受）
   → 用储备金购买抵押品、核销债务

3. 协议金库（协议承受）
   → 用金库资产填补缺口

4. 治理代币稀释（代币持有者承受）
   → 增发代币，用销售收入弥补

5. 存款人承担（用户承受，最后手段）
   → 直接减少存款人的余额
```

### 坏账处理的自动化

```move
// 坏账处理流程
fun handle_bad_debt(
    system: &mut CDPSystem,
    reserve: &mut Reserve,
    bad_debt_amount: u64,
) {
    // 第一优先级：使用储备金
    let reserve_balance = reserve.balance();
    if (reserve_balance >= bad_debt_amount) {
        reserve.withdraw(bad_debt_amount);
        system.write_off(bad_debt_amount);
        return;
    };

    // 第二优先级：部分使用储备金，剩余由治理决定
    reserve.withdraw(reserve_balance);
    system.write_off(reserve_balance);
    let remaining = bad_debt_amount - reserve_balance;

    // 触发治理投票处理剩余坏账
    emit_bad_debt_event(remaining);
}
```

## 链上保险

### 保险基金模型

```
保险基金运作模型：

投保人：支付保费 → 保险基金
索赔人：发生损失 → 提交索赔 → 保险基金赔付

保费定价：
  年化保费 = 保额 × 风险率
  风险率取决于：
    - 协议审计状态
    - TVL 大小
    - 历史索赔记录
    - 资产类型
```

### Sui 上的保险实现

Sui 的对象模型使得保险合约可以实现得更清晰：

```
InsurancePolicy 对象：
  - 保额
  - 保费
  - 保险期
  - 理赔条件
  - 状态（有效/已索赔/已过期）

索赔流程：
  1. 被保事件发生（如协议被攻击）
  2. 索赔人提交证据
  3. 预言机/DAO 验证事件
  4. 保险基金赔付
```

### 保险基金的挑战

| 挑战 | 说明 |
|------|------|
| 预言机问题 | 如何确定"损失确实发生了"？ |
| 道德风险 | 被保险协议是否因此降低安全标准？ |
| 逆选择 | 最需要保险的协议风险也最高 |
| 资金效率 | 保险基金的资金利用率通常很低 |

## 储备金充足率评估

如何判断储备金是否充足？

```
储备金充足率 = 储备金余额 / 最大可能坏账

最大可能坏账的估算方法：
  = Σ (每个仓位的潜在坏账)
  = Σ max(0, 债务 - 抵押品 × 最低可能价格)

最低可能价格可以根据历史最大回撤来估算：
  - BTC 历史最大回撤: ~80%
  - ETH 历史最大回撤: ~85%
  - 山寨币历史最大回撤: ~95%

建议储备金充足率 > 50%（覆盖一半以上的极端情况坏账）
```

## 实践建议

1. **多级储备**：每种资产独立设置储备金，避免交叉补贴
2. **定期评估**：每月重新评估储备金充足率
3. **自动化处理**：坏账处理逻辑写在合约中，不需要治理投票
4. **透明报告**：定期发布储备金状态报告
5. **压力测试**：模拟极端市场条件下储备金的消耗速度
