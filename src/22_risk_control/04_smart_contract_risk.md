# 22.4 智能合约风险模型

第 18 章详细分析了具体的攻击手法（预言机操纵、闪电贷攻击、重入等）。本节从风险管理者的视角，建立一个**系统性的合约风险分类模型**——不是讨论"怎么攻击"，而是讨论"如何系统性地识别和评估风险"。

## 风险分类：STRIDE 模型适配 DeFi

借鉴信息安全领域的 STRIDE 模型，适配 DeFi 合约的具体场景：

| 风险类别                               | DeFi 中的表现                | 可利用性         |
| -------------------------------------- | ---------------------------- | ---------------- |
| **S**poofing（身份伪造）               | 冒充管理员、伪造预言机数据源 | 需要绕过权限检查 |
| **T**ampering（数据篡改）              | 修改价格、余额、状态变量     | 通常需要权限漏洞 |
| **R**epudiation（抵赖）                | 链上不可篡改，此类风险低     | —                |
| **I**nformation disclosure（信息泄露） | 交易排序信息被 MEV 利用      | 公开链上数据     |
| **D**enial of Service（拒绝服务）      | 阻塞清算、耗尽 Gas           | 低成本可实施     |
| **E**levation of Privilege（提权）     | 普通用户执行管理操作         | 最严重的风险     |

## 资金流向分析法

评估合约风险最有效的方法是**追踪资金流向**：

```
用户资金流入 → 协议处理 → 资金流出

关键问题：
1. 资金在协议中的每一跳是否安全？
2. 是否存在资金可以被"截留"的路径？
3. 资金流出的条件是否正确？
4. 是否有资金被永久锁定的风险？
```

对每个资金操作建立**风险矩阵**：

| 操作 | 风险             | 触发条件       | 影响         | 缓释措施            |
| ---- | ---------------- | -------------- | ------------ | ------------------- |
| 存款 | 存入金额记录错误 | 精度问题       | 用户损失     | 使用 `Balance` 类型 |
| 借款 | 超额借款         | 抵押率计算错误 | 协议坏账     | 保守的 LTV          |
| 清算 | 清算金额不足     | 计算逻辑错误   | 坏账累积     | Close Factor 限制   |
| 提款 | 无力支付         | 利用率过高     | 用户资金锁定 | 流动性储备          |
| 转账 | 发送错误         | 地址验证缺失   | 资金丢失     | 类型安全            |

## 外部交互风险

合约与外部系统的交互是风险最集中的地方：

### 1. 预言机交互

```move
// 安全模式：校验价格合理性
let price = oracle::price(&oracle);
assert!(price >= MIN_PRICE && price <= MAX_PRICE, EInvalidPrice);
```

风险点：

- 预言机返回过期数据
- 预言机返回极端值
- 预言机合约本身被攻击

### 2. 外部代币交互

```move
// 风险：不同精度的代币

```

## 风险量化：影响 × 可能性

对识别出的风险进行量化评估：

| 等级 | 影响                   | 可能性             | 风险值      |
| ---- | ---------------------- | ------------------ | ----------- |
| 关键 | 资金永久损失           | 高（已知攻击模式） | 🔴 立即处理 |
| 高   | 大额资金暂时锁定       | 中（需要特定条件） | 🟠 优先处理 |
| 中   | 小额资金损失或功能异常 | 中                 | 🟡 计划处理 |
| 低   | 用户体验问题           | 高                 | 🟢 酌情处理 |

## Move 相对于 EVM 的风险降低

| 风险类别 | EVM (Solidity)                  | Sui Move               | 风险降低      |
| -------- | ------------------------------- | ---------------------- | ------------- |
| 重入     | 高风险，需要手动防护            | 资源模型天然防御       | ✅ 基本消除   |
| 双花     | 需要手动检查                    | 类型系统保证           | ✅ 基本消除   |
| 整数溢出 | Solidity 0.8 前高风险           | 默认检查               | ✅ 编译器保证 |
| 权限混乱 | `msg.sender` / `tx.origin` 混淆 | Capability 对象        | ✅ 结构化     |
| 逻辑漏洞 | 相同风险                        | 相同风险               | ⚠️ 未消除     |
| 升级风险 | Proxy 模式复杂                  | 包升级更简洁但仍有风险 | ⚠️ 部分降低   |

关键认知：Move 消除了很多经典漏洞类别，但**逻辑漏洞和设计缺陷仍然是最主要的风险来源**。这正是审计和形式化验证的价值所在。
