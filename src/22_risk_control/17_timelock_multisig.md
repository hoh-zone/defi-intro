# 22.17 Timelock、多签与紧急权限

治理安全依赖于三个核心机制：时间锁（Timelock）、多签（Multisig）和紧急权限（Emergency Authority）。它们构成了协议安全的"行政管理体系"。

## Timelock（时间锁）

### 原理

时间锁的核心思想：**所有关键操作的执行都强制延迟**，给社区和利益相关者审查和反应的时间。

```
没有时间锁：
  治理投票通过 → 立即执行
  → 如果是恶意提案，来不及反应

有时间锁：
  治理投票通过 → 等待 48 小时 → 执行
  → 社区有 48 小时发现问题并取消
```

### Timelock 的类型

| 类型 | 描述 | 适用场景 |
|------|------|---------|
| 固定延迟 | 所有操作统一延迟 | 简单协议 |
| 分级延迟 | 不同风险等级不同延迟 | 推荐方案 |
| 自适应延迟 | 根据操作影响动态调整 | 高级协议 |

### 分级延迟设计

```
操作分类与延迟：

低风险操作（延迟 24 小时）：
  - 调整利率曲线参数（小幅）
  - 添加新的市场/资产
  - 更新预言机地址

中风险操作（延迟 48 小时）：
  - 修改清算参数
  - 修改最大 LTV
  - 增加储备因子

高风险操作（延迟 7 天）：
  - 合约升级
  - 金库资金转移
  - 修改治理参数本身
```

### Sui 上的 Timelock 实现

Sui 的 epoch 机制天然适合实现时间锁：

```move
// 使用 epoch 作为时间锁
public struct TimelockedProposal has key {
    id: UID,
    target_epoch: u64,      // 最早执行 epoch
    executed: bool,
}

fun execute_after_timelock(
    proposal: &mut TimelockedProposal,
    ctx: &TxContext,
) {
    assert!(ctx.epoch() >= proposal.target_epoch, ETimelockNotExpired);
    assert!(!proposal.executed, EAlreadyExecuted);
    proposal.executed = true;
    // 执行提案内容...
}
```

## 多签（Multisig）

### 原理

多签要求 N 个批准者中有 M 个同意才能执行操作（M-of-N）。

```
单签（危险）：
  1 个私钥泄露 → 攻击者可以执行任何操作

多签（安全）：
  3-of-5 多签 → 需要至少 3 个私钥同时泄露
  攻击难度显著提高
```

### 多签参数选择

| 方案 | 安全性 | 效率 | 适用场景 |
|------|--------|------|---------|
| 2-of-3 | 中 | 高 | 小型协议 |
| 3-of-5 | 高 | 中 | 推荐 |
| 4-of-7 | 很高 | 低 | 大型协议 |
| 5-of-9 | 极高 | 很低 | 基础设施 |

### 多签持有者的选择标准

1. **独立性**：签名人之间没有利益关联
2. **专业性**：理解协议的技术和金融风险
3. **多样性**：来自不同组织/地域
4. **可追溯性**：每次签名都有记录和理由
5. **可替换性**：有机制替换不活跃的签名人

### Sui 多签实现

Sui 原生支持多签地址：

```
Sui 多签地址 = 参与 pubkey 列表 + 阈值

创建：
  sui keytool multisig add --threshold 3
    --pk addr1 --pk addr2 --pk addr3
    --pk addr4 --pk addr5

特点：
  - 链原生支持，无需合约
  - 可与 PTB 结合
  - 支持不同权重
```

## 紧急权限

### Emergency Guardian

紧急权限是在极端情况下快速响应的机制。与常规治理不同，紧急权限可以在短时间内执行关键操作。

```
正常治理流程：
  提案 → 投票（3-7天）→ 时间锁（1-3天）→ 执行
  总耗时: 4-10 天

紧急流程：
  多签批准（小时级）→ 短时间锁（2-12小时）→ 执行
  总耗时: 2-24 小时
```

### 紧急权限的范围

| 操作 | 常规治理 | 紧急权限 |
|------|---------|---------|
| 暂停协议 | ✗（太慢） | ✅ |
| 调整清算参数 | ✗（太慢） | ✅ |
| 修改利率曲线 | ✅ | ⚠️ 有限调整 |
| 转移资金 | ✅ | ✗（禁止） |
| 合约升级 | ✅ | ✗（禁止） |
| 取消紧急操作 | ✅ | ✅（需要更高级别） |

关键原则：**紧急权限只能"关"不能"开"**——可以暂停功能，但不能添加新功能或转移资金。

### Circuit Breaker（熔断器）

```
熔断器设计：

触发条件：
  - 价格 30 分钟内变化 > 30%
  - 利用率 > 95%
  - 清算失败率 > 20%
  - 大额异常提款（> TVL 的 5% 在 1 小时内）

触发后的操作：
  1. 暂停新的借款
  2. 暂停新的存款（防止用户在不知情时存入）
  3. 清算继续（保护协议偿付能力）
  4. 提款继续（保护用户退出权）
  5. 通知社区和监控

恢复条件：
  - 触发条件消失
  - 或通过治理投票手动恢复
```

## 权限层级设计

```
┌──────────────────────────────────────────┐
│            权限层级架构                    │
├──────────────────────────────────────────┤
│ 第 0 层：不可变层                         │
│   核心合约逻辑（不可升级/不可修改）        │
├──────────────────────────────────────────┤
│ 第 1 层：治理层                           │
│   参数调整、功能升级（需要代币投票）       │
│   时间锁: 1-7 天                          │
├──────────────────────────────────────────┤
│ 第 2 层：紧急层                           │
│   暂停/恢复、紧急参数调整（需要多签）      │
│   时间锁: 2-12 小时                       │
├──────────────────────────────────────────┤
│ 第 3 层：自动层                           │
│   熔断器触发、利率自动调整（无需人工）     │
│   无时间锁（自动执行）                     │
└──────────────────────────────────────────┘
```

这个层级设计确保：
- 日常操作通过治理投票（去中心化、透明）
- 紧急情况通过多签快速响应（效率与安全平衡）
- 系统级保护通过自动化实现（无人工依赖）
- 核心逻辑不可变（最高安全级别）
