# 9.6 算法稳定币：机制谱系、历史教训与教学用代码

## 「算法」指什么

在本书语境下，**算法稳定币**指：**主要依赖规则（供给调节、债券、rebase、多代币博弈等）而非足额链上抵押**来维持锚定的一类设计。实务中常与「部分抵押」「治理铸币权」混合出现。

常见机制标签（概念层）：

| 类型 | 思路 | 备注 |
|------|------|------|
| Rebase | 调节每个地址余额数量以保持目标单价 | 实现与 UX 复杂 |
| Seigniorage / 债券 | 溢价时扩张、折价时发债吸收供应 | 依赖持续需求 |
| 双币 / 三币模型 | 稳定币 + 权益 / 债券代币博弈 | 典型死亡螺旋风险 |
| 纯算法（无外部抵押） | 仅靠预期与套利闭环 | 高压下多次失败 |

## 历史教训（为什么本书不把它当作主实现）

公开市场上，**无足额抵押的纯算法锚定**在极端行情中多次崩溃（流动性枯竭、信心断裂、正反馈抛售）。从风险工程角度，应默认：

- **机制在链上可写 ≠ 经济在长期可续**；
- 压力测试必须包含「抵押品暴跌 + 用户挤兑 + 预言机异常」的组合场景（见第 **22** 章）。

本书**不以算法稳定币作为唯一主代码路径**，而是用极小模块展示「供给与规则状态如何落在链上」，与 **9.5** 的 CDP 形成对比。

## 教学包：`algorithmic_stablecoin_sketch`

路径：`src/09_stablecoin/code/algorithmic_stablecoin_sketch/`

模块 **`algorithmic_stablecoin_sketch::algo`**（一次性见证 **`ALGO`**）包含：

- **`AlgoEngine`**：记录**名义供给** `nominal_supply`、**准备金** `reserve`（示意性的部分准备）、以及简易 **peg 分子/分母**。
- **`governance_expand`**：治理增加名义供给与准备金（类比扩张阶段、发债注水——仅示意）。
- **`contraction_burn`**：收缩名义供给并减少准备金（示意销毁/赎回）。

**刻意省略**：真实 AMM、用户持有的 `Coin`、预言机、债券拍卖、清算与任何「可持续锚定证明」。读者应把本模块理解为 **状态机速写**，而非可上线协议。

### 读代码时自问

1. 若 `nominal_supply` 与真实 `Coin` 总供应量脱钩，系统是否还诚实？
2. 扩张权力集中在 `GovernanceCap` 时，与法币型的 `IssuerCap` 有何异同？
3. 若去掉所有抵押品与清算，仅靠规则调节，**最后一道信任落在哪里**？

## 与 CDP 的边界

- **CDP（9.4–9.5）**：债务与抵押有明确链上对应关系，清算提供最后防线。
- **算法（本节）**：代码可以极简，但**经济约束**极难补齐——故生产级设计往往走向「部分抵押 + 算法」混合，并接受更重的审计与参数治理。
