# 9.5 从机制正确到系统可信度

## 机制正确 ≠ 系统可信

你的 CDP 合约可以通过所有测试，利率模型参数合理，清算逻辑正确。但这还不够。

**可信度（Credibility）** 是用户愿意把钱放在你的协议里的理由。它包含六个层面：

### 代码可信度

- 合约是否经过审计？
- 审计方是谁？声誉如何？
- 是否有 bug bounty 计划？
- 代码是否开源？

### 资产可信度

- 抵押品是否是成熟资产？
- 稳定币是否有足够的使用场景？
- 是否有赎回机制？

### 预言机可信度

- 使用什么预言机？
- 有没有多源验证？
- 预言机失效时协议如何响应？

### 参数可信度

- 参数是如何确定的？
- 参数修改的流程是什么？
- 是否有时间锁（timelock）？

### 治理可信度

- 谁持有治理权限？
- 治理决策是否透明？
- 是否有渐进式去中心化的路径？

### 运营可信度

- 团队是否有处理紧急情况的经验？
- 是否有应急预案？
- 是否有保险机制？

## 透明性的角色

透明性不是"把代码放在 GitHub 上"。真正的透明性是：
- 参数变化的理由被记录和解释
- 每次清算的数据可以被验证
- 系统状态（总债务、抵押品价值、健康分布）实时可查

## 反例场景

假设一个 CDP 协议：
- 代码经过审计 ✓
- 使用 Pyth 预言机 ✓
- 参数设计合理 ✓

但是：
- 治理密钥由单人持有 ✗
- 参数修改没有时间锁 ✗
- 没有紧急暂停机制 ✗
- 稳定币没有实际使用场景 ✗

这个协议的机制是正确的，但系统不可信。治理密钥持有者可以修改参数为自己谋利，稳定币因为没有使用场景而缺乏需求。

**机制正确是必要条件，系统可信是充分条件。** 两者缺一不可。
