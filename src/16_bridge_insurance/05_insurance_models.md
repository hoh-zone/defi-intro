# 16.5 链上保险的原理与模型

## 为什么需要链上保险

```
DeFi 协议的风险清单：
  - 智能合约漏洞
  - 预言机操纵
  - 治理攻击
  - 清算失败
  - 跨链桥攻击
  - 稳定币脱锚

传统金融有存款保险、FDIC、对冲工具。
DeFi 需要自己的保险机制。
```

## 三种链上保险模型

### 模型 1：参数型保险（Parametric）

```
原理：
  预定义触发条件（参数）
  条件满足时自动赔付
  不需要人工评估

示例：
  触发条件：稳定币价格 < $0.95 持续 1 小时
  赔付金额：投保金额 × (1 - 当前价格)
  理赔过程：链上价格预言机自动触发

优势：
  理赔快速（自动化）
  无争议（条件清晰）
  透明（规则链上可查）

劣势：
  只能覆盖可量化的风险
  预言机本身是风险点
  难以覆盖复杂事件（如"合约被黑"）
```

### 模型 2：互助型保险（Mutual）

```
原理：
  成员共同出资形成资金池
  成员遭受损失时从池中赔付
  成员通过投票决定是否赔付

示例（Nexus Mutual 模式）：
  1. 成员质押 NXM 代币加入互助池
  2. 用户购买保单（用 ETH 支付保费）
  3. 出事时提交索赔申请
  4. 其他成员投票决定是否赔付
  5. 赔付从互助池中支出

优势：
  可以覆盖任意类型的风险
  社区驱动，去中心化

劣势：
  理赔流程慢（需要投票）
  可能出现拒赔（投票不通过）
  资金池规模限制承保能力
```

### 模型 3：预测市场型保险

```
原理：
  创建一个二元预测市场："协议 X 在 Y 时间内会被攻击吗？"
  "是"代币和"否"代币的价格反映市场对风险的评估
  保单 = 购买"是"代币（赌协议会出事）

示例：
  协议 A 的"会被攻击"代币价格 = $0.05
  → 市场认为 5% 的概率被攻击
  用户花 $5 购买面值 $100 的"会被攻击"代币
  如果协议被攻击：用户获得 $100（赔付）
  如果协议安全：用户损失 $5（保费）

优势：
  市场定价，无需精算模型
  可以覆盖任何可定义的事件
  流动性提供者赚取保费

劣势：
  需要足够流动性
  预测市场本身可能被操纵
  事件定义的歧义可能导致争议
```

## 保费定价

```
保费 = 预期损失 × 风险调整系数

预期损失 = 保额 × 概率(事件发生) × 损失率

示例：
  保额：$10,000
  被攻击概率（历史数据）：5%/年
  损失率（被攻击时的平均损失）：80%

  预期损失 = $10,000 × 5% × 80% = $400
  风险调整系数 = 1.5（覆盖模型不确定性）
  年保费 = $400 × 1.5 = $600（6% 费率）
```

## 三种模型对比

| 维度 | 参数型 | 互助型 | 预测市场型 |
|---|---|---|---|
| 理赔速度 | 即时 | 慢（投票） | 即时（市场结算） |
| 覆盖范围 | 窄（可量化） | 宽（任意） | 宽（可定义） |
| 争议风险 | 低 | 中（投票争议） | 高（事件定义） |
| 承保能力 | 取决于资金池 | 取决于互助池 | 取决于流动性 |
| 去中心化程度 | 中 | 高 | 高 |
| 代表协议 | Etherisc | Nexus Mutual | Polymarket |

## 风险分析

| 风险 | 描述 |
|---|---|
| 系统性风险 | 多个协议同时出事，保险池被抽干 |
| 对手方风险 | 保险协议本身被攻击 |
| 预言机风险 | 参数型保险的触发条件依赖预言机 |
| 道德风险 | 投保后用户可能降低安全意识 |
| 逆向选择 | 高风险用户更愿意购买保险 |
