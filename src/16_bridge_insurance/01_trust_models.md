# 16.1 跨链桥的信任模型与分类

## 为什么需要跨链桥

```
问题：ETH 在以太坊上，SUI 在 Sui 上。如何在两条链之间转移资产？

答案：你不能真正"转移"——ETH 是以太坊的原生资产，
     它不存在于 Sui 上。你只能做一个"等价交换"。

跨链桥的本质：在源链锁定资产，在目标链发行代表凭证。
```

## 四种跨链桥架构

### 1. 锁铸桥（Lock-and-Mint）

```
源链（以太坊）          目标链（Sui）
┌──────────────┐        ┌──────────────┐
│  Lock 100 ETH│──验证──│ Mint 100 wrappedETH
│  在金库合约  │        │ 在 Sui 上    │
└──────────────┘        └──────────────┘

赎回：在 Sui 上销毁 wrappedETH → 以太坊上释放 ETH

信任假设：金库合约和验证者的安全
代表：Wormhole（原始模式）、Multichain
```

### 2. 轻客户端桥（Light Client）

```
源链（以太坊）          目标链（Sui）
┌──────────────┐        ┌──────────────┐
│  发送交易    │        │ 以太坊轻客户端│
│  包含证明    │──证明──│ 验证区块头   │
│              │        │ 确认后执行   │
└──────────────┘        └──────────────┘

信任假设：目标链上的轻客户端正确验证源链共识
代表：IBC（Cosmos 生态）、Sui 原生桥（对以太坊）
优势：安全性最高（不引入额外信任假设）
劣势：实现复杂、验证成本高
```

### 3. 中继桥（Relay）

```
源链（以太坊）          目标链（Sui）
┌──────────────┐        ┌──────────────┐
│  发送事件    │        │ 接收并验证   │
│              │←中继──→│              │
│  中继者监听  │        │ 中继者提交   │
└──────────────┘        └──────────────┘

信任假设：中继者的诚实性（通常需要多签或质押）
代表：Axelar、LayerZero
```

### 4. 乐观桥（Optimistic）

```
源链（以太坊）          目标链（Sui）
┌──────────────┐        ┌──────────────┐
│  发送消息    │        │ 先执行，后验证│
│              │───→    │ 挑战期窗口   │
│  任何人可挑战│←───    │ 无人挑战则确认│
└──────────────┘        └──────────────┘

信任假设：至少有一个诚实的挑战者（1-of-N）
代表：Optimism Bridge、Nomad（已废弃）
优势：信任假设极弱
劣势：延迟高（需要等待挑战期）
```

## 信任模型对比

```
安全性排名（从高到低）：

1. 轻客户端桥：安全性 = min(源链, 目标链)
   → 不引入新的信任假设

2. 乐观桥：安全性 = 至少 1 个诚实挑战者
   → 1-of-N 信任假设

3. 中继桥：安全性 = 中继者集合
   → M-of-N 信任假设（通常是 2/3 多数）

4. 锁铸桥：安全性 = 金库控制者
   → 可能是 5-of-8 多签（如 Ronin 被攻破时）
```

## 跨链桥的核心组件

```
┌─────────────────────────────────────┐
│            跨链桥架构                │
├─────────────────────────────────────┤
│  源链合约                            │
│  ├─ Lock：锁定资产                   │
│  ├─ Emit Event：发出跨链消息         │
│  └─ Verify：验证目标链的证明         │
│                                     │
│  中继层                              │
│  ├─ Watcher：监听源链事件            │
│  ├─ Attestor：签名确认事件           │
│  └─ Relayer：提交到目标链            │
│                                     │
│  目标链合约                          │
│  ├─ Mint：铸造代表资产               │
│  ├─ Verify Proof：验证源链证明       │
│  └─ Execute：执行跨链调用            │
└─────────────────────────────────────┘
```

## 跨链消息的类型

```
类型 1：资产转移
  锁定 ETH → 铸造 wrapped ETH
  最简单，最常见

类型 2：消息传递
  源链发消息 → 目标链执行
  允许跨链合约调用

类型 3：组合调用
  源链调用 → 目标链调用 → 第三链调用
  失败时需要整条链回滚
```

## 风险分析

| 桥类型 | 已知攻击              | 损失   |
| ------ | --------------------- | ------ |
| 锁铸桥 | Ronin 多签被攻破      | $624M  |
| 锁铸桥 | Wormhole 签名验证漏洞 | $326M  |
| 乐观桥 | Nomad 初始化漏洞      | $190M  |
| 中继桥 | Multichain 密钥泄露   | ~$130M |
| 各类型 | Harmony Horizon       | $100M  |

20 亿美元的教训：**跨链桥的安全瓶颈不是区块链本身，而是桥的实现**。
