# 20.1 审计前的文档与自查清单

## 审计需要提供的材料

### 1. 架构文档

描述协议的整体结构、模块关系、数据流。

```
应包含：
- 模块依赖图
- 对象生命周期图
- 关键函数调用流程
- 外部依赖（预言机、其他协议）
```

好的架构文档能让审计师在 **4 小时内**理解你的协议。差的文档会让审计师花 2 天梳理结构——而你在按天付费。

一份合格的架构文档应包含：

**模块关系图**：每个 Move 模块的职责和调用关系。
```
market.move ──调用──> math.move
    │                     │
    ├──调用──> oracle.move │
    │                     │
    └──调用──> share.move ┘
```

**对象生命周期**：每个核心对象从创建到销毁的完整路径。
```
SupplyPool 创建于 init()
  → 用户 deposit() 获得 SupplyPosition
  → SupplyPosition 可被 transfer
  → withdraw() 时 SupplyPosition 被 destroy
  → SupplyPool 在协议暂停时可冻结
```

**外部依赖清单**：预言机、其他协议合约、链上常量。

### 2. 流程图

每个核心操作的流程图（存款、借款、清算等）。

```
存款流程：
用户调用 deposit() → 检查暂停状态 → 计算份额 → 更新池子余额 → 铸造 Position → 发出事件
```

流程图应标注**失败路径**，不仅是 happy path：

```
存款流程（含失败路径）：
用户调用 deposit()
  → [暂停?] → 是 → abort EPaused
  → [金额 > 0?] → 否 → abort EInvalidAmount
  → [达到存款上限?] → 是 → abort EDepositCapReached
  → 计算份额
  → [份额 > 0?] → 否 → abort EShareTooSmall
  → 更新池子余额 → 铸造 Position → 发出事件
```

### 3. 风险文档

列出所有已知的风险点和对应措施。

```
| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 预言机操纵 | 不当清算 | TWAP + 多源验证 |
| 清算者缺席 | 坏账累积 | 动态激励 + 保险基金 |
| 管理员恶意 | 参数操纵 | 多签 + 时间锁 |
```

风险文档还应包含**攻击场景描述**：每个风险的具体攻击路径、前提条件、影响范围。审计师会基于这些场景设计对抗测试。

### 4. 测试报告

测试覆盖率报告和对抗测试清单。

审计师需要看到的不只是覆盖率数字，而是**测试清单**：

```
正常路径测试：
- deposit → withdraw 完整流程
- borrow → repay 完整流程
- 多用户并发存取

边界条件测试：
- 存入最小金额
- 利用率 0% 和 100% 的利率
- 健康因子刚好在清算线

对抗测试：
- 预言机返回极端价格
- 清算时抵押品价值为 0
- 同一交易内 deposit + withdraw（闪电贷场景）
- 管理员参数修改后立即触发清算
```

### 5. 参数表

所有可配置参数的当前值、合理范围、修改权限。

| 参数                      | 当前值 | 范围        | 修改权限            |
| ------------------------- | ------ | ----------- | ------------------- |
| collateral_ratio_bps      | 15000  | 12000-20000 | AdminCap + Timelock |
| liquidation_threshold_bps | 13000  | 11000-18000 | AdminCap + Timelock |
| fee_bps                   | 30     | 0-100       | AdminCap            |

参数表还应说明**参数之间的约束关系**：

```
约束：
- MCR > LR > 100%（开仓阈值 > 清算阈值 > 100%）
- fee_bps < max_fee_bps（手续费不能超过上限）
- debt_ceiling > current_debt（债务上限不能低于当前债务）
```

### 6. 事件和错误码文档

列出所有事件和错误码的含义。

事件文档示例：

```move
// 当用户存款时发出
public struct Deposit has copy, drop {
    user: address,      // 存款用户
    amount: u64,        // 存入金额
    shares: u64,        // 获得份额
    exchange_rate: u64, // 当前汇率
}
```

错误码文档示例：

```move
const EPaused: u64 = 0;           // 协议已暂停
const EInvalidAmount: u64 = 1;    // 无效金额（0 或溢出）
const EDepositCapReached: u64 = 2; // 存款上限已达
```

审计师会用事件文档验证**状态变更的可观测性**，用错误码文档验证**错误处理的完整性**。

## 文档准备的时间线

```
审计开始前 2 周：
  - 完成架构文档和流程图
  - 完成参数表和事件文档
  - 内部自查清单通过

审计开始前 1 周：
  - 完成风险文档和攻击场景
  - 完成测试报告
  - 部署到测试网，提供交互指引

审计开始：
  - 提交所有文档 + 代码仓库访问 + 测试网部署信息
```

## 九项自查清单

1. 所有 public 函数都有权限检查或明确标注"无需权限"
2. 所有状态修改都发出了事件
3. 所有除法都处理了除零情况
4. 所有中间计算使用了 u128
5. 所有外部数据（预言机）都有校验
6. 所有 AdminCap 的使用都有对应的事件记录
7. 所有 struct 的 ability 组合是有意为之
8. 所有测试都包含正常和异常路径
9. 部署脚本在测试网验证过
