## 上线不是终点

上线只是从"开发模式"切换到"运营模式"。新的风险会持续出现：

- 市场极端行情
- 新的攻击手法
- 依赖的协议出问题
- 链本身的升级或故障

**上线不是终点，而是监控的起点。**

## 上线检查流程

```
阶段 1：内部验证（上线前 2 周）
  ├── 所有测试在 testnet 通过
  ├── 参数表经过压力测试模拟
  ├── 暂停/恢复流程演练
  └── 团队每人独立完成一次完整操作

阶段 2：安全审计（上线前 1 周）
  ├── 收到审计报告
  ├── 处理所有 Critical 和 High 级别发现
  ├── Medium 级别有明确的接受或修复计划
  └── 修复后的代码重新审计

阶段 3：最终确认（上线前 1 天）
  ├── 主网部署脚本 dry run
  ├── AdminCap 转移到多签地址
  ├── 时间锁参数确认
  ├── 监控和告警系统就绪
  └── 紧急响应联系人确认

阶段 4：上线（当日）
  ├── 部署合约
  ├── 验证合约字节码
  ├── 初始化协议参数
  ├── 转移 AdminCap 到多签
  ├── 启动监控
  └── 宣布上线

阶段 5：上线后（第一周）
  ├── 密切监控所有指标
  ├── 准备紧急暂停
  ├── 收集社区反馈
  └── 处理紧急问题
```

## 主网部署验证清单

部署完成后，在主网上逐项验证：

```move
// 1. 验证包 ID 和模块
sui client call --package <PACKAGE_ID> --module <MODULE> ...

// 2. 验证 AdminCap 持有者
sui client object <ADMIN_CAP_ID> --owner

// 3. 验证初始参数
sui client object <PROTOCOL_SYSTEM_ID>
// 检查：paused = false, 参数值 = 预期值

// 4. 执行一笔测试交易
sui client call --package <PACKAGE_ID> --module market --function deposit ...

// 5. 验证事件发出
sui client events --package <PACKAGE_ID>
```

## 上线后的持续审计

上线并不意味着审计结束：

| 时间节点   | 动作                                 |
| ---------- | ------------------------------------ |
| 上线 24h   | 检查所有事件是否正常、无异常 abort   |
| 上线 1 周  | 第一次参数审查，确认是否需要调整     |
| 上线 1 月  | 回顾风险文档，更新已知风险           |
| 每次升级   | 差异审计：只审计变更部分             |
| 重大事件后 | 事后审计：分析事件并检查防御是否有效 |
