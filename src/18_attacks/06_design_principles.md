# 18.6 从攻击模式中提炼设计准则

## 十条安全设计原则

### 1. 不信任单一价格源

使用 TWAP + 多源验证 + 偏差阈值。单点价格在任何时候都可能被操纵。

### 2. 每个状态修改函数都要检查权限

在 Sui 上通过 AdminCap 参数实现。忘记检查权限 = 给所有人开了管理员后门。

### 3. 先验证，后执行

所有前置条件检查必须在状态修改之前完成。检查和修改之间不能有可失败的中间操作。

### 4. 用 u128 做中间计算

`u64` 的最大值是 18.4 * 10^18。两个 u64 相乘很容易溢出。所有乘法先用 u128，最后再转回 u64。

### 5. 考虑零值和极值

传入 0 会怎样？传入 u64::MAX 会怎样？每个 public 函数都要测试这两个边界。

### 6. 对象分离：配置与资金分开

预言机配置、风险参数等经常更新的对象，不应该和存储资金的对象混在一起。更新配置时不应该阻塞资金操作。

### 7. 清算机制要有经济激励

清算不是自动的。如果没有人为清算有利润，清算就不会发生。动态调整清算罚金，确保在 Gas 高时仍然有激励。

### 8. 有紧急暂停开关

极端情况下，协议需要能快速暂停所有操作。暂停权限应该通过多签 + 时间锁保护。

### 9. 所有事件都要记录

状态变化时发出事件。事件是链下监控的基础。没有事件 = 黑盒操作。

```move
public fun deposit(pool: &mut Pool, amount: u64, ctx: &mut TxContext) {
    sui::event::emit(DepositEvent {
        pool_id: object::id(pool),
        depositor: ctx.sender(),
        amount,
        timestamp: sui::clock::timestamp_ms(sui::clock::create_for_testing()),
    });
}
```

### 10. 测试对抗场景，不只是正常流程

单元测试覆盖"存 100 取 100"是不够的。你需要测试：
- 存 0 会怎样？
- 存 u64::MAX 会怎样？
- 同一笔交易中存入后立即取出会怎样？
- 预言机价格突然变成 0 会怎样？
- 管理员在用户存款时暂停协议会怎样？

## 总结

| 原则 | 对应的攻击类型 |
|------|---------------|
| 1. 不信任单一价格源 | 预言机操纵 |
| 2. 检查权限 | 权限滥用 |
| 3. 先验证后执行 | 逻辑漏洞 |
| 4. u128 中间计算 | 整数溢出 |
| 5. 考虑零值和极值 | 边界条件攻击 |
| 6. 对象分离 | DoS / Gas 竞争 |
| 7. 清算经济激励 | 清算失效 |
| 8. 紧急暂停 | 级联崩盘 |
| 9. 事件记录 | 无法事后分析 |
| 10. 对抗测试 | 所有攻击类型 |

每一条规则背后都是一次真实的损失。不是理论，是教训。
